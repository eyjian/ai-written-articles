# Superpowers 不等于 Harness Engineering：一个偏工作流，一个偏控制面

## 先把结论说白

如果你赶时间，可以先记这三句：

- **`Superpowers`**：更接近 AI 编码场景里的工作流实践，重点是把需求澄清、计划拆解、TDD、调试、审查、验证这些环节补齐。
- **`Harness Engineering`**：更接近围绕 Agent 构建控制面、反馈机制和运行环境的一种工程方法。

## `Harness Engineering` 这个词，原始语境是什么

目前公开讨论里，一个重要源头是 Mitchell Hashimoto 在 2026 年 2 月 5 日发表的《My AI Adoption Journey》。他当时的说法其实很谨慎：行业里未必已经有一个被广泛接受的正式术语，但他倾向把这类工作叫作 `harness engineering`。

他想表达的核心意思并不复杂：**当你发现 Agent 会反复犯某类错时，不要只补一次 prompt，而是把解决方案工程化，让它以后尽量别再犯同样的错。** 这可能表现为维护 `AGENTS.md`、补验证脚本、加测试、加工具，或者把高风险步骤变成可重复执行的流程。

到了 2026 年 4 月 2 日，Birgitta Böckeler 在 Martin Fowler 网站发表《Harness engineering for coding agent users》，把这套思路整理得更系统，也提出了一个很有启发的观察角度：**要建立对 coding agent 的信任，不能只靠 prompt，还要靠前馈式引导和反馈式传感。**

很多后续文章喜欢把这种思路压缩成一句话：

> **Agent ≈ Model + Harness**

这句话很好记，也确实有解释力。但要说准一点：**它更像社区后来对这套思路的概括，不是 Mitchell 原文里直接写下的公式。**

## 用一把更顺手的尺子看：前馈、反馈、长期治理

如果借 Birgitta Böckeler 那篇文章里的思路来看，可以把这类控制机制先粗略分成两部分：

- **Feedforward Guides**：Agent 动手之前，先给它方向、边界和约束
- **Feedback Sensors**：Agent 动手之后，用各种机制检查它有没有偏航

如果再往长期一点看，还会多出第三层问题：**系统跑久以后，文档漂不漂、技术债积不积、规则会不会失效。** 这已经更接近治理层，而不只是单次任务里的引导或纠偏。

用这把尺子回头看 `Superpowers`，会比较顺：

| 观察维度 | `Harness Engineering` 更关心什么 | `Superpowers` 的典型表现 |
|---------|----------------------------------|---------------------------|
| **前馈** | 让 Agent 开工前就拿到足够背景、边界和规则 | 有覆盖，比如需求澄清、计划拆解、把约束提前说透 |
| **反馈** | 在执行过程中尽快发现错误并拉回正轨 | 覆盖很强，这恰好是 `Superpowers` 最有价值的部分 |
| **长期治理** | 处理文档漂移、技术债、系统性偏差 | 通常不是它的核心强项，往往需要额外机制补足 |

这样看，你就会明白为什么很多人会觉得两者“很像”：因为 `Superpowers` 确实抓住了 `Harness` 视角里很关键的一段，尤其是**前馈 + 强反馈回路**这一段。

但也正因为如此，更准确的说法不是：
“`Superpowers` = `Harness Engineering`。”

而是：**`Superpowers` 更像把一部分前馈机制和一条很强的反馈回路，压缩成了一套适合 AI 编码场景直接上手的工作流实践。**

这也是为什么很多人第一次上手 `Superpowers`，会明显感觉 AI 没那么“毛躁”了。通常不是模型突然更聪明了，而是**它被放进了一条更讲纪律的工作流里。**
