# Agent 框架里哪些假设该删了：从"我可以停止做什么"重新审视工程设计

> 搭 Agent 系统的团队几乎都经历过一个阶段：框架代码越写越多，编排逻辑越来越复杂，但 Agent 的表现并没有线性提升——有时甚至在退步。问题出在哪？不是功能不够，而是假设太多。Anthropic 在 Claude Code 的工程实践中提出了一个反直觉的问题："Ask: what can I stop doing?" 不是问框架还缺什么，而是问框架里哪些东西可以拿掉了。

## 框架越写越厚，不是因为功能多了，而是假设堆多了

Agent 框架的复杂度增长，很大一部分不是功能需求驱动的，而是"不信任模型"的假设驱动的。每个假设都对应一层额外的框架代码，每层代码都带来一组新的配置项和一种新的失败模式。

这些假设的来源很典型：早期模型工具调用不稳定，于是加一层结果校验；模型不会自己决定调用顺序，于是搭编排引擎；模型上下文管理差，于是手工往 prompt 里塞满指令；模型不会主动记录关键信息，于是围绕它搭一整套外部记忆系统。

每个假设在提出时都有道理。但假设会堆叠——框架每多一层假设，就多一层中间代码、多一组配置、多一种需要排查的故障链路。到最后，框架本身变成了系统里最大的复杂度来源。

问题在于：这些假设当初是对的，但模型在进化。当短板被模型自身补上时，为弥补短板而设计的框架层就变成了死重（dead weight）。

| 旧假设 | 对应的框架层 | 催生的工程代价 | 当前判断 |
|--------|------------|--------------|---------|
| 模型不会自己选工具调用顺序 | 编排引擎（DAG / 状态机） | 编排代码量大，每加一个工具都要改流程图 | 需要重新评估 |
| 每个工具结果都必须流回上下文窗口 | 结果收集与注入层 | token 消耗高，延迟增加 | 可以放手 |
| 模型不会管理自己的上下文 | 手工 prompt 填塞 + 强制压缩 | 注意力预算被挤占，每加一条指令都稀释效果 | 部分可以放手 |
| 模型不会主动做持久化决策 | 外部 RAG 检索管线 | 检索架构复杂，维护成本高 | 需要重新评估 |

这张表不是说所有框架层都该拆掉，而是说每一层都值得重新问一个问题：**这个假设在当前模型能力下还成立吗？**

## 三种需要重新审视的假设

### 编排自主权——框架还需要替 Agent 排工序吗

传统 Agent 框架有一个根深蒂固的假设：模型不会自己编排工具调用，所以必须由框架来排工序——用 DAG 定义流程、用状态机管理转换、用编排引擎控制执行顺序。这在早期是务实的选择，但代价也很明显：每增加一个工具、每修改一条路径，都要改编排逻辑。

Anthropic 的实测数据给出了另一个视角。

在 SWE-bench Verified 基准测试中，Claude 3.5 Sonnet 仅用 bash 和文本编辑器两个基础工具就达到了 49% 的解题率——这在当时是最高水平。没有 DAG，没有状态机，没有编排引擎，模型自己决定什么时候读文件、什么时候改代码、什么时候跑测试。基础工具组合已经足够，关键在于模型能否自主决定怎么用。

更直接的证据来自 BrowseComp 基准。当给 Opus 4.6 过滤自身工具输出的能力后——也就是让模型自己决定哪些中间结果需要保留、哪些可以丢弃——准确率从 45.3% 提升到 61.6%。这不是小幅改善，是超过 16 个百分点的跳升。编排权从框架交给模型，效果反而更好。

背后的逻辑不复杂：框架硬编排的问题在于，它只能按预设路径走。遇到预期外的工具输出，编排引擎要么走兜底分支（如果有的话），要么报错退出。而模型自主编排的优势在于，它能根据实际的中间结果动态调整下一步——这本质上是一个实时决策问题，模型比静态流程图更适合处理。

工程上的落地方式也在变化。与其搭一个复杂的编排引擎，不如给模型一个代码执行工具（bash 或 REPL），让模型自己写代码串联调用。模型自己决定哪些中间结果需要看、哪些可以跳过，避免所有工具输出都灌回上下文。编排引擎不是不需要了，而是从"替 Agent 排工序"退化为"提供工序能力"——框架负责暴露工具接口，模型负责决定怎么组合。

但放手有边界：

- **可以放手的**：工具调用的具体顺序、中间结果的过滤和管道
- **仍然不能放手的**：安全边界（哪些工具不能调）、失败后的升级路径、成本上限

编排细节可以交给模型，但"栏杆"必须留在框架里。

### 上下文管理权——框架还需要替 Agent 管工作台吗

另一个长期存在的假设是：模型不会管理自己的上下文，所以框架必须替它做——在系统提示词里塞满手工指令，预加载大量背景文档，手动控制什么信息进入上下文窗口。

这种做法的问题在哪？上下文窗口是有限的注意力预算。每多塞一条指令，留给当前任务的空间就少一点。当指令堆到一定密度，模型对每条指令的关注度反而下降——相当于给一个人同时塞了 30 条工作注意事项，结果一条都记不住。

Anthropic 在实践中发现，模型已经具备三种上下文自管理能力。

**第一种是按需加载。** Claude Code 的 Skills 机制只在系统提示词中预加载短描述（YAML frontmatter），完整内容在模型判断需要时才从文件中读取。这比一股脑把所有 Skill 内容塞进上下文要高效得多——模型自己知道什么时候需要什么信息。

**第二种是上下文编辑。** 模型可以选择性地删除上下文中过时的工具结果或已经不再相关的思考块。这相当于自己整理工作台——把用过的草稿纸清理掉，只保留当前任务需要的材料。

**第三种是子代理分叉。** 当任务需要隔离上下文时，模型可以启动一个子代理，在独立的上下文窗口中处理特定子任务，完成后只把结果带回主上下文。在 BrowseComp 基准上，子代理能力比最佳单代理运行额外提升了 2.8%——模型知道什么时候应该分叉新上下文。

一个典型案例揭示了假设过期的代价。Sonnet 4.5 在长任务中会出现"上下文焦虑"——感知到窗口快满了就过早结束任务。工程团队为此加了上下文重置机制，在合适的时机清空上下文重新开始。到了 Opus 4.5，这个问题不再出现了——模型自身就能处理长上下文。但之前为 Sonnet 4.5 加的重置机制还在，变成了新模型的死重。如果不及时清理，框架里就会积累越来越多这种"为旧模型打的补丁"。

放手与兜底的边界：

- **可以放手的**：上下文的填充策略、过时信息的清理、子任务的隔离决策
- **仍然不能放手的**：安全敏感信息的注入控制（哪些信息不能进入上下文）、上下文总量的硬上限

### 持久化策略——框架还需要替 Agent 管记忆吗

第三个假设是：模型不会自己做持久化决策，所以框架必须搭 RAG 检索基础设施——向量数据库、embedding 管线、检索策略、重排序模型——由框架决定存什么、什么时候检索、怎么注入上下文。

这套基础设施的工程成本不低：需要选型和维护向量数据库，需要设计 embedding 策略，需要调优检索参数，需要处理检索结果与当前上下文的融合。每一环都是可能出问题的地方。

但 Anthropic 的测试数据显示，模型的持久化能力正在快速增长。

在 BrowseComp 的压缩测试中——也就是评估模型在历史上下文被压缩后还能保持多少性能——三代模型的表现差异巨大：Sonnet 4.5 无论给多少压缩预算都保持在 43% 左右，几乎没有利用压缩信息的能力；Opus 4.5 扩展到 68%；Opus 4.6 达到 84%。压缩能力不是框架给的，是模型自带的，而且在快速进步。

更有意思的数据来自记忆文件夹实验。在 BrowseComp-Plus 任务中，给 Sonnet 4.5 一个可以自由读写的记忆文件夹，准确率从 60.4% 提升到 67.2%。没有向量数据库，没有 embedding 管线，没有检索策略——就是最朴素的文件读写，模型自己决定往文件里写什么、什么时候回头查阅。

宝可梦游戏的对比测试把两代模型的记忆行为差异展示得更加直观：

| 维度 | Sonnet 3.5 | Opus 4.6 |
|------|-----------|----------|
| 14000 步后文件数 | 31 个（含重复） | 10 个（有目录组织） |
| 进度 | 仍在第二个城镇 | 获得三个徽章 |
| 记忆风格 | 抄录员——记录 NPC 说了什么 | 战术家——提炼自身失败经验，记录可复用策略 |
| 关键差异 | 没有从记忆中提取可行动信息 | 有专门的 learnings 文件，记录如"Bellsprout Sleep+Wrap combo"这样的具体战术 |

Sonnet 3.5 像一个不会做笔记的学生——把老师讲的每句话都抄下来，但从不回头看，也不提炼重点。Opus 4.6 则像一个有经验的工程师——只记关键决策和失败教训，用结构化的方式组织，需要时能快速定位。

工程含义很清晰：简单的文件读写（记忆文件夹）比复杂的向量数据库 + RAG 管线更适合当前阶段。不是"不需要记忆系统"，而是记忆系统从"框架驱动"转向"模型驱动 + 框架提供存储能力"。

放手与兜底的边界：

- **可以放手的**：记忆内容的选取和组织方式、检索时机的决策
- **仍然不能放手的**：存储容量的限制、敏感信息的脱敏、记忆的生命周期管理

## 放手不等于撤退——Harness 从编排转向边界

拿掉旧假设不代表 Harness 变得不重要。恰恰相反，Harness 的重心需要迁移——从"替 Agent 做决策"转向"给 Agent 划边界"。

这里有三个常见误读需要先澄清。

**误区一："模型强了就不需要 Harness 了。"** 这是最容易踩的坑。放手的是编排细节、上下文填充和记忆检索这些"决策类"职责。安全边界、质量兜底、成本控制这些"约束类"职责不仅不能放手，反而因为模型自主权增大而变得更重要——自主权越大，围栏就越得牢固。

**误区二："所有假设都该拿掉。"** 安全类假设永远不能放手。沙箱隔离、权限控制、成本上限——这些不是因为模型不够聪明才存在的，而是因为系统运行本身就需要边界。无论模型多强，"不能删生产数据库"这条约束都得编进框架里。

**误区三："放手可以一步到位。"** 不能。每个假设的拿掉都需要逐个评估、灰度验证、可回滚。一次性拆掉所有编排逻辑然后"看看会怎样"，这不是工程决策，是赌博。

Harness 的重心迁移方向是从控制到能力供给：

```text
旧模式（Harness 重心 = 编排控制）：
┌─────────────────────────────────────┐
│  编排引擎 │ 上下文填塞 │ RAG 检索   │  ← 框架替 Agent 做决策
├─────────────────────────────────────┤
│  约束 │ 沙箱 │ 反馈回路 │ 可观测性    │  ← 框架给 Agent 划边界
└─────────────────────────────────────┘

新模式（Harness 重心 = 边界控制）：
┌─────────────────────────────────────┐
│  约束 │ 沙箱 │ 反馈回路 │ 可观测性    │  ← 框架给 Agent 划边界（不变且更重要）
├─────────────────────────────────────┤
│  工具供给 │ 存储能力 │ 成本限制      │  ← 框架提供能力，Agent 自行决策
├─────────────────────────────────────┤
│  Agent 自主编排 / 上下文自管理 / 记忆自组织 │
└─────────────────────────────────────┘
```

旧模式里，框架最厚的一层是编排控制；新模式里，最厚的一层是边界控制。框架的代码量可能减少了，但每一行的分量更重了。

还有两件事仍然是框架的活，不该交给模型。

**缓存优化**是其中之一。静态内容放前面、动态内容放后面、不轻易换模型、谨慎管理工具列表的更新断点——这些是 token 成本和响应延迟的优化，属于框架的 UX 和成本责任。原因很直接：API 的提示词缓存按前缀匹配，前缀不变才能命中缓存，一旦靠前的内容频繁变动，后面所有 token 的缓存全部失效。模型自己无法优化自己的缓存命中率。

**声明式工具**是另一件。将需要门控的操作提升为专用工具（而不是靠 prompt 里写"请不要执行危险命令"），用工具的参数定义来编码安全边界和用户确认流程。Claude Code 的自动模式就是一个例子：当主 Agent 想执行一条 shell 命令时，框架会先把这条命令交给第二个独立的 Claude 实例做安全评估——这个实例只负责判断"这条命令能不能放行"，通过了才真正执行。换句话说，安全判断不再是提示词里的一条建议，而是架构中由独立实例把关的硬约束。这类约束永远属于框架，不属于模型。

## 一张判断清单：这个假设该拿掉还是留着

不是所有假设都该拿掉，也不是所有假设都该留着。需要一个结构化的判断框架来逐个评估。

| 判断维度 | 可以放手的信号 | 仍需兜底的信号 |
|---------|--------------|--------------|
| 安全性 | 操作仅影响沙箱内环境 | 涉及生产数据、敏感信息、不可逆操作 |
| 可验证性 | 结果可被自动化验证（测试、lint、类型检查） | 结果只能靠人工判断质量 |
| 成本可控性 | 单次失败的成本可接受（重试即可） | 失败成本高昂或不可回滚 |
| 模型能力匹配 | 当前模型在该任务上已有基准数据支撑 | 没有数据支撑，仅凭经验假设 |
| 降级路径 | 放手后有清晰的监控 + 人工接管路径 | 放手后无法及时发现问题 |

这张表的使用方式是：拿框架中任何一个"替 Agent 做决策"的模块，逐列对照。五个维度全部亮绿灯才考虑放手；任何一个亮红灯就暂时保留。

具体落地分三步。

**第一步，盘点。** 列出框架中所有"替 Agent 做决策"的代码，标记每段代码对应的假设。这一步的产出是一张"假设清单"——大多数团队在做完盘点后会发现，自己框架里替模型做的决策比想象中多得多。

**第二步，评估。** 用上面的决策表逐个评估每个假设是否仍然成立。重点关注"模型能力匹配"这一列——如果已经有基准测试数据说明模型在该任务上表现足够好，那这个假设就是放手的候选项。

**第三步，灰度。** 不要一次性拿掉。先在非关键路径上试——比如先在开发环境放开编排自主权，监控模型的工具调用序列和成功率。确认稳定后再逐步推广到生产路径。每次放手都要有回滚方案。

## 收尾

回到开头的现象：框架越写越厚，Agent 的表现反而没有提升。根本原因不是功能不够，而是旧假设没有及时清理。为 Sonnet 4.5 加的上下文重置在 Opus 4.5 上变成了死重，为早期模型搭的编排引擎在模型能自主编排后变成了多余的中间层，为不会做笔记的模型搭的 RAG 管线在模型学会自己写 learnings 文件后变成了过度工程。

"问自己什么可以停止做"不是偷懒，而是一种工程纪律。每当模型升级一个版本，就应该重新过一遍假设清单：哪些框架层还在解决真实问题？哪些已经变成了为旧短板打的补丁？

这个判断过程本身是持续的。今天可以放手的假设，明天可能因为业务场景变化又需要重新兜底；今天必须兜底的假设，下一代模型可能就能自己搞定。Harness 要素系列文章讲的是"该管什么"——约束、沙箱、反馈回路、上下文管理、可观测性。本篇补上的是"该停止管什么"。两者合在一起，才是完整的 Harness 工程判断：知道该在哪里加围栏，也知道该在哪里清理死重。
