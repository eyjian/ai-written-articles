# Agent 的生产基础设施该自建还是托管：从 Managed Agents 看 Build vs Buy 的工程决策

> Anthropic 发布了 Claude Managed Agents——安全沙箱、身份认证、会话持久化、多 Agent 协调、执行追踪，打包成一套托管 API。Notion、Rakuten、Asana、Sentry 已经接入。这不是又一个模型能力升级的故事，而是一个老问题换了新皮：Agent 的生产基础设施，该自建还是托管？

## Agent 上生产缺的不是模型，是基础设施

一个 Agent Demo 跑通了，能调工具、能对话、能生成结果。团队很兴奋，决定推到生产环境。然后翻车开始了。

Agent 在沙箱里跑得好好的，到了生产环境直接操作了线上数据库——没有隔离机制，工具调用和生产资源之间没有任何屏障。一个长任务跑到一半，网络抖动，会话断了，中间状态全部丢失，只能从头再来。出了 bug 想排查，才发现 Agent 的每一步决策没有日志，根本不知道它在哪一步走偏的。

这三个问题没有一个跟模型能力有关。模型够聪明，工具够丰富，prompt 也调好了——但生产环境需要的不是这些。

把 Agent 从 Demo 推到生产，需要的基础设施清单远比想象中长：

| 基础设施层 | 要解决的问题 | 自建需要做什么 | Managed Agents 提供了什么 |
|-----------|------------|--------------|------------------------|
| 安全沙箱 | Agent 执行不能碰生产环境 | 容器隔离、网络策略、文件系统限制 | 生产级安全沙箱 |
| 身份与权限 | 谁能调、能调什么 | OAuth/RBAC/API Key 管理 | 范围权限 + 身份管理 |
| 会话持久化 | 长任务不能断了就没了 | 状态存储、断点续跑、会话恢复 | 长时间运行会话 |
| 多 Agent 协调 | 多个 Agent 怎么分工不打架 | 编排引擎、消息队列、状态同步 | 多 Agent 协调 |
| 可观测性 | 出了问题能查到是哪一步 | 日志、trace、指标采集 | 执行追踪 |
| 自动验证 | Agent 说"做完了"不能光信它 | 测试框架、lint、端到端验证 | 自我评估与迭代 |

这张表里的每一行，都是一个独立的工程子系统。自建的话，每个子系统都需要选型、开发、测试、运维。六行加起来，工程量不亚于搭一套微服务治理平台。

这和十年前微服务从单体拆分时的处境很像。服务拆出来不难，难的是拆出来之后的治理——服务发现、负载均衡、熔断降级、链路追踪、配置中心。当年很多团队低估了治理的工程量，结果服务拆完了，治理跟不上，系统比单体时期还脆弱。Agent 上生产也是同一个剧本：模型调通了只是开头，基础设施才是真正的工程量。

## Managed Agents 到底在卖什么

Managed Agents 不是一个更强的模型，不是一个新的 API 端点，也不是一套 Agent 框架。它不提升模型能力——Claude 该什么水平还是什么水平。

它卖的是运维负担的转移。

上一节列的六层基础设施，每一层自建都有两笔成本：初始搭建成本和持续运维成本。Managed Agents 做的事情是把这两笔成本打包，换成一个可预测的价格：标准 token 费率 + 每活跃会话小时 $0.08。（定价信息以 Anthropic 官网为准，本文数据基于发布时的公开信息。）

$0.08 这个数字值得拆一下。它收的不是模型推理的费用——那部分已经算在 token 费率里了。这 $0.08 是基础设施租金：沙箱的计算资源、会话状态的存储、断连恢复的机制、执行追踪的日志系统。换句话说，Managed Agents 的定价模型本身就在暗示一件事——Agent 的生产成本里，基础设施占据了独立的一块。

据 Anthropic 官方数据，相比标准提示循环，Managed Agents 的任务成功率提高多达 10 个百分点。这 10 个百分点不是模型变聪明了。同一个模型，同样的 prompt，放在标准 API 调用里和放在 Managed Agents 里，差距来自哪里？来自环境因素导致的失败减少了——会话断了能恢复、工具执行有沙箱兜底、长任务有持久化保障。模型该犯的推理错误一个没少，但基础设施层面的失败被消除了。

几个客户案例可以快速说明适用场景：

- **Notion**：长时间运行会话和记忆管理。典型的长任务场景，Agent 需要在多轮交互中保持上下文。
- **Rakuten**：企业级 Agent 一周内完成部署。如果自建基础设施，一周时间连安全评审都过不完。
- **Asana**：AI Teammates，在项目管理工具内直接分配任务给 Agent。多 Agent 协调的落地场景。
- **Sentry**：调试 + 修复。Agent 需要读日志、定位问题、生成修复——每一步都需要可观测性支撑。
- **Atlassian**：Jira 内直接分配任务。类似 Asana，Agent 作为团队成员被编排进已有工作流。

这些案例有一个共同特征：都是在已有产品中嵌入 Agent 能力，而不是从零搭一个 Agent 产品。对这类场景来说，自建基础设施的性价比很低——核心产品是 Notion、Jira、Sentry，不是基础设施本身。

什么时候该关注 Managed Agents？团队的核心价值在应用层而不是基础设施层的时候。反过来，如果 Agent 还在原型验证阶段，连需求都没稳定，这时候锁定任何平台都太早。

## 自建的真实成本——比想象中贵

自建 Agent 基础设施的成本有一个经典特征：看得见的部分远不是全部。

**看得见的成本：**

搭一套最小可用的生产级基础设施——沙箱隔离、身份认证、会话持久化、基本的日志追踪——初始开发周期在 2-3 个月。需要 2-3 个有分布式系统经验的工程师全职投入。这部分成本大多数团队能估到。

**看不见的成本才是大头：**

| 成本类型 | 看得见的 | 看不见的 |
|---------|---------|---------|
| 人力 | 初始开发 2-3 人月 | 持续运维每月占 15-25% 工程师时间 |
| 安全 | 基本的沙箱隔离 | 安全漏洞修补、权限模型迭代、合规审计 |
| 可靠性 | 首次部署能跑 | 断连恢复、状态一致性、故障自愈 |
| 规模 | 小规模验证通过 | 并发扩容、资源调度、性能调优 |
| 知识 | 搭建期的技术攻关 | 核心工程师离职后的知识断层 |

15-25% 这个数字是关键。意思是搭完之后，团队每个月要拿出六分之一到四分之一的工程师时间来维护这套基础设施。这不是偶发的修 bug，而是持续的运维负担——安全补丁、版本升级、容量规划、监控报警调优。

还有一笔经常被遗漏的成本：机会成本。花在基础设施上的工程师时间，原本可以花在产品功能上。对于一个 10 人团队，每月 15-25% 的运维投入意味着 1.5-2.5 人月的产能被基础设施吃掉。一年下来，相当于少了一到两个工程师的全年产出。

这和"要不要自建 Kubernetes 集群"是同一类决策。2016 年前后，很多团队选择自建 K8s 集群——自己搭控制面、自己管 etcd、自己维护网络插件。后来大部分团队迁到了托管的 EKS、GKE、AKS。不是自建不行，而是基础设施的运维负担会随着规模增长而指数膨胀，大多数团队的核心竞争力不在这里。

自建有一个隐含前提：团队有能力长期维护。如果只看初始搭建就做决策，大概率会低估总成本。

## 托管的代价——便宜的背面

托管解决了运维负担，但引入了另一组问题。

**厂商锁定**是最直接的代价。一旦业务逻辑建立在 Managed Agents 的会话管理、多 Agent 协调、执行追踪之上，迁移成本就开始累积。这些不是标准化的接口——每个厂商的实现方式不同，迁移意味着重写。锁定的深度取决于使用的功能深度：只用基础的 API 调用，锁定浅；用了多 Agent 协调和会话持久化，锁定深。

**可控性下降**也是实际问题。自建系统出了问题，团队可以直接查代码、加日志、改逻辑。托管服务出了问题，只能提工单等响应。当 Agent 在生产环境出现异常行为，排查路径的长短直接影响修复速度。托管服务提供了执行追踪，但追踪的粒度和深度由服务商决定，不一定够用。

**定价风险**需要用具体数字来感受。以发布时的公开定价 $0.08 每活跃会话小时为例：

- 1000 个并发会话 × 8 小时/天 × $0.08 = $640/天
- 一个月按 30 天算：$640 × 30 = $19,200/月
- 一年：$19,200 × 12 = $230,400/年

这还只是基础设施租金，不含 token 费用。$19,200/月足够雇一个全职基础设施工程师。当规模到达一定量级，托管的成本曲线会和自建交叉——小规模时托管便宜，大规模时自建可能更划算。

当然，这只是粗算。实际业务中，并发会话数不是恒定的，高峰低谷差异可能很大。但这组数字揭示了一个规律：**托管服务的成本和规模线性相关，自建的成本和规模非线性相关。** 小规模时，自建的固定成本被摊薄不了；大规模时，托管的线性成本反而成为负担。

**数据主权**是第四个代价。Agent 的执行过程中会接触业务数据——用户信息、内部文档、代码仓库。这些数据在托管环境中的存储位置、访问权限、生命周期，都由服务商的策略决定。对于强监管行业（金融、医疗、政务），数据不能出境或不能交给第三方处理，这一条就可能直接否决托管方案。

## 判断框架——什么时候自建、什么时候托管、什么时候混合

Build vs Buy 不是一个凭直觉拍板的决策。需要一个可复用的判断矩阵，把模糊的"感觉"转化为可对照的维度。

**决策矩阵：**

| 判断维度 | 倾向托管 | 倾向自建 |
|---------|---------|---------|
| 团队规模 | < 10 人，没有专职基础设施团队 | 有专职平台工程团队 |
| Agent 成熟度 | 还在验证阶段，需要快速试错（前提：方向大概率会持续，否则锁定任何平台都太早） | 已经跑通，进入规模化阶段 |
| 数据敏感度 | 公开数据或低敏感度 | 强监管行业、数据不能出境 |
| 规模预期 | 小规模（< 100 并发会话） | 大规模（> 1000 并发会话） |
| 定制需求 | 标准功能够用 | 需要深度定制编排逻辑或安全策略 |
| 迁移风险容忍度 | 能接受一定程度的厂商锁定 | 必须保持厂商中立 |

使用方式：逐行对照，数一下"倾向托管"和"倾向自建"各有几个。全部或大部分在同一侧，决策清晰；两侧各半，考虑混合策略。

混合策略怎么落地？核心思路是**按层拆分**：

把基础设施拆成"通用层"和"业务层"。通用层——沙箱隔离、会话持久化、基础日志——用托管服务。这些是标准化程度高、运维负担重、但业务差异化低的部分。业务层——自定义的多 Agent 编排逻辑、特定行业的合规策略、深度定制的执行追踪——自建。这些是业务差异化的核心，也是竞争壁垒所在。

```text
混合架构示意：

┌──────────────────────────────────┐
│  业务层（自建）                    │
│  自定义编排 │ 行业合规 │ 定制追踪  │
├──────────────────────────────────┤
│  抽象接口层                       │
│  适配器模式，隔离上下层依赖         │
├──────────────────────────────────┤
│  通用层（托管）                    │
│  沙箱 │ 会话持久化 │ 身份认证     │
└──────────────────────────────────┘
```

中间的抽象接口层是关键。它的作用是：让业务层不直接依赖托管服务的具体 API，而是依赖自定义的抽象接口。日后如果要从 Managed Agents 迁移到其他方案，或者从托管切回自建，只需要替换抽象层下面的实现，业务层不用动。

什么时候该用混合策略？两个信号：一是团队规模中等（10-50 人），有一定基础设施能力但不想全押在运维上；二是业务中既有标准化需求又有深度定制需求。满足其中一个，混合就值得考虑。

什么时候别用混合策略？团队规模太小（< 5 人），同时维护自建和托管两套体系的复杂度反而更高。这种情况下，选一边，先跑起来。

## 今天选的那一边，半年后可能要翻

上面的决策矩阵有一个前提：它是基于当前状态做的判断。但"当前状态"会变。

一个典型的翻转路径：团队初期只有 5 个人，Agent 还在验证阶段，用 Managed Agents 快速上线。半年后验证跑通了，业务规模上来，并发会话从 50 涨到 2000。按之前的数字算，$0.08 × 2000 × 8 × 30 = $38,400/月。与此同时团队也扩到了 30 人，有了专职的平台工程小组。这时候，决策矩阵里的每一行指向都可能从"托管"翻转到"自建"。

反方向的翻转也存在。一个团队花三个月自建了全套基础设施，结果 Agent 业务方向调整，并发量没上来，三个工程师的运维投入变成了沉没成本。早知如此，不如一开始就用托管。

Build vs Buy 不是一次性决策，而是需要定期重新评估的持续过程。降低翻转代价的办法不是选对一次，而是让每次切换都不至于伤筋动骨。

**保留抽象接口**是最基本的一条。无论选自建还是托管，业务逻辑不要直接耦合到具体的基础设施实现上。用适配器模式隔离依赖——今天调的是 Managed Agents 的 API，明天可能换成自建的服务，抽象层不变就不用改业务代码。

在此基础上，**定期重新评估**比"选对一次"更重要。建议每个季度对照决策矩阵重新打分。团队规模变了吗？业务规模变了吗？数据合规要求变了吗？托管服务的定价调整了吗？任何一个维度的变化都可能触发决策翻转。但光有评估还不够，还需要**把决策流程固化下来**：什么指标触发重新评估、谁来做评估、评估结论怎么落地。写进团队的季度复盘中，变成例行动作而不是临时起意。

抽象接口降低切换的技术成本，定期评估降低切换的信息滞后，固化流程降低切换的组织摩擦。三件事做到了，无论当前选了哪一边，翻转的代价都在可控范围内。
