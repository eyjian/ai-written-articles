# QMD 的混合搜索架构：本地 RAG 搜索引擎是怎么做到又快又准的

> 上了 RAG 之后，很多团队发现搜索质量不稳定：精确命令查不到、语义搜索混进无关内容、排序靠向量距离根本不可靠。问题不在向量搜索本身，而在于只有向量搜索。QMD 的做法是把 BM25 全文搜索、向量语义搜索、LLM 重排序三种引擎融合在一起，全跑在本地——这套混合搜索管线的工程设计，值得拆开看。

## QMD 是什么

QMD 是 Shopify 创始人 Tobi Lütke 的开源项目，GitHub 上 21.3k stars，MIT 协议。

定位很明确：为个人知识库和 Agent 工作流设计的本地搜索引擎。一句话概括——BM25 + 向量搜索 + LLM 重排序，全部本地运行，无需云 API。

需要先划清边界：QMD 不是 Elasticsearch 的替代品。它不做分布式，不做多租户，不做大规模集群。它解决的是个人和小团队的知识管理搜索问题——把本地的 Markdown 文档和代码文件索引起来，用混合搜索找到最相关的内容。

## 纯向量搜索的三个短板

在拆 QMD 的架构之前，先说清楚一个前提判断：为什么只靠向量搜索不够。

### 精确匹配盲区

搜 `qmd embed --force` 这样的精确命令，向量搜索可能返回一堆讲"嵌入"概念的文档。因为向量模型把 "embed" 编码成了语义空间里"嵌入"的含义，而不是把它当作一个精确的命令字符串。越是技术文档、命令行参数、API 名称这类精确查询，向量搜索越容易跑偏。

### 语义漂移

查询和文档的语义空间不完全对齐。模型训练时学到的语义关联，和具体领域的实际关联经常对不上。比如搜"怎么解决内存泄漏"，可能召回一篇讲"垃圾回收机制概述"的文档——语义上确实相关，但实际上不解决问题。这种"看起来相关但其实不对"的结果，在 RAG 里是最难处理的噪声。

### 缺少最终相关性判断

向量搜索的排序完全靠向量距离。距离近 ≠ 真的回答了问题。一段文档可能和查询在语义空间里很近，但它讲的是背景知识而不是直接答案。纯向量搜索缺少一个最终的"裁判"来判断：这篇到底回答了问题没有？

三个短板叠在一起：精确查不到、语义查不准、排序没判断——这就是为什么很多 RAG 系统上线后召回质量不稳定。

## QMD 的三层搜索引擎

QMD 的解法是用三种搜索方式互补，每种覆盖其他两种的盲区。

| 搜索方式 | 引擎 | 擅长 | 不擅长 |
|---------|------|-----|--------|
| BM25 全文搜索 | SQLite FTS5 | 精确关键词匹配、特定术语、命令名 | 不理解语义近义词 |
| 向量语义搜索 | embeddinggemma-300M | 语义相似、跨表述匹配 | 精确字面匹配、短查询 |
| LLM 重排序 | qwen3-reranker-0.6B | 理解查询意图，判断相关性 | 需要候选集，不能独立检索 |

BM25 解决精确匹配，向量搜索解决语义理解，LLM 重排序解决最终相关性判断。三者的关系不是替代，而是管线——先用前两种召回候选集，再用第三种做最终排序。

值得注意的是引擎选型。向量搜索用的是 embeddinggemma-300M，一个 300M 参数的小模型，本地跑没有压力。重排序用的是 qwen3-reranker-0.6B，600M 参数，同样本地运行。全链路不依赖任何云 API，这是 QMD "本地优先"设计理念的直接体现。

## 混合搜索管线的工程设计

这是 QMD 最值得拆的部分。`qmd query` 命令背后是一条 6 步管线，每一步都有明确的工程考量。

### Step 1：查询扩展

用微调的 1.7B 模型生成查询变体。比如原始查询是"怎么嵌入 Markdown 文件"，模型可能生成"qmd embed 命令""索引 Markdown 文档"等扩展查询。

关键设计：原始查询给 ×2 权重。

为什么？因为查询扩展是把"一个问题"变成"几个相关问题"来增加召回率，但扩展查询不一定比原始查询更准。如果不给原始查询加权，精确匹配结果很容易被扩展查询带来的噪声稀释掉。×2 权重是一个经验值，保证原始查询的精确匹配不会被淹没。

### Step 2：多路并行检索

每个查询（原始 + 扩展）分别走 BM25 和向量搜索两条路。

假设查询扩展生成了 2 个变体，加上原始查询一共 3 个查询，每个走 2 条搜索路径：

> 3 个查询 × 2 种搜索 = 6 路结果

6 路结果各有侧重——BM25 路径捞精确匹配，向量路径捞语义相关。多路并行是为了保证召回的覆盖面，宁可多召回一些再排序，也不能在召回阶段就漏掉关键文档。

### Step 3：RRF 融合

6 路结果需要合并成一个排序列表。QMD 用的是 Reciprocal Rank Fusion（RRF），参数 k=60。

RRF 的核心公式：

```text
score = Σ(1 / (k + rank + 1))
```

直觉解释：一个文档如果在多条路径里排名都靠前，它的 RRF 分数就高。不管是靠精确匹配排上来的，还是靠语义相关排上来的，只要多条路径都认为它重要，最终得分就高。

这里同样有权重设计：原始查询结果给 ×2 权重。和 Step 1 的逻辑一致——保护原始查询的精确匹配结果。

为什么选 RRF 而不是简单地合并分数？因为 BM25 分数和向量距离的量纲完全不同，直接加减没有意义。RRF 只看排名不看分数，天然解决了跨模态分数不可比的问题。

### Step 4：Top-Rank Bonus

在任意一路检索中排名 #1 的文档额外 +0.05 分，排名 #2-3 的文档 +0.02 分。

这个设计解决什么问题？假设一个文档在 BM25 路径里排名 #1（精确匹配最佳），但在其他路径里排名一般。经过 RRF 融合后，它的总分可能被那些"每条路径都中等偏上但没有一条路径特别靠前"的文档超过。

Top-Rank Bonus 就是给"在某条路径里绝对领先"的文档一个保护机制——如果 BM25 认定这是精确匹配最好的结果，不能因为语义搜索那边的排名一般就把它挤掉。

### Step 5：LLM 重排序

取 RRF + Top-Rank Bonus 之后的 Top 30 候选文档，送入 qwen3-reranker-0.6B 做重排序。

重排序的方式很直接：对每个候选文档，让 reranker 回答"这个文档是否回答了查询的问题"，输出 yes/no，并取 logprobs 作为置信度得分。

这一步补上了纯向量搜索缺少的"最终相关性判断"。前面的 BM25 和向量搜索解决的是"哪些文档可能相关"，reranker 解决的是"哪些文档真正回答了问题"。

### Step 6：位置感知混合

最后一步是把 RRF 分数和 Reranker 分数按位置混合：

| 排名区间 | RRF 权重 | Reranker 权重 | 设计意图 |
|---------|---------|-------------|---------|
| Rank 1-3 | 75% | 25% | 保护精确匹配 |
| Rank 4-10 | 60% | 40% | 平衡两者 |
| Rank 11+ | 40% | 60% | 后排更信任重排序 |

为什么不直接用 Reranker 的分数做最终排序？

因为 Reranker 有一个已知问题：它可能把 RRF 阶段排名很高的精确匹配给"降权"掉。Reranker 是基于语义理解判断相关性的，它可能认为一篇"全面介绍相关概念"的文档比一篇"精确回答问题但内容很短"的文档更相关。但在搜索场景里，精确匹配往往才是最需要的结果。

位置感知混合的逻辑是：**前排结果更信任 RRF（因为精确匹配大概率在前排），后排结果更信任 Reranker（因为后排的区分度更依赖语义理解）**。这是一个工程上的折中方案，比"二选一"更稳定。

### 管线总览

```text
原始查询
  │
  ├─→ 查询扩展（1.7B 模型，生成 2 个变体）
  │     原始查询 ×2 权重
  │
  ├─→ 多路并行检索（3 查询 × 2 路径 = 6 路结果）
  │     BM25（SQLite FTS5）+ 向量搜索（embeddinggemma-300M）
  │
  ├─→ RRF 融合（k=60，原始查询 ×2 权重）
  │
  ├─→ Top-Rank Bonus（#1: +0.05, #2-3: +0.02）
  │
  ├─→ LLM 重排序（Top 30 → qwen3-reranker-0.6B）
  │
  └─→ 位置感知混合（前排信 RRF，后排信 Reranker）
```

6 步管线里有一个贯穿始终的设计意图：**保护精确匹配**。从查询扩展的 ×2 权重，到 RRF 的 ×2 权重，到 Top-Rank Bonus，到位置感知混合的前排 75% RRF——每一步都在防止精确匹配被"语义相关但其实不对"的结果挤掉。这说明 QMD 团队在实践中吃过这个亏，才会在架构设计上反复加固。

## 智能分块：搜得准的前提是切得对

搜索引擎再好，如果文档被切得支离破碎，召回的也是残缺信息。QMD 在分块策略上做了几件值得注意的事。

### Markdown 结构评分分块

不是按固定字符数切，而是按 Markdown 结构评分来决定切分点：

| 结构元素 | 分数 | 含义 |
|---------|------|------|
| 标题（#、##、###） | 100 | 最强切分点 |
| 代码块边界（```） | 80 | 次强切分点 |
| 段落分隔（空行） | 20 | 弱切分点 |

评分越高的位置越优先作为切分边界。这意味着标题之间的内容会尽量保持完整，不会出现"一段话被切成两半"的情况。

### 代码围栏保护

代码块内部不切割。不管代码块有多长，都会被当作一个完整单元保留。这一条看似简单，但很多分块工具做不到——按字符数切的方案很容易把一个函数从中间劈开，导致检索出来的代码片段无法理解。

### AST 感知分块

对代码文件，QMD 提供了一个可选的 AST 感知分块模式：用 tree-sitter 解析代码的语法树，在函数、类、import 语句的边界处切分。

这意味着分块结果是"一个完整的函数""一组 import 语句""一个类定义"，而不是"某个函数的前半段 + 上一个函数的后半段"。

### 分块参数

默认配置：每块约 900 tokens，块与块之间 15% 重叠。

15% 重叠是为了处理切分边界处的信息——如果某个关键内容恰好在切分点附近，重叠保证它至少完整地出现在一个块里。

## 为 Agent 而生的设计

QMD 不只是一个搜索引擎，它从一开始就为 Agent 工作流做了适配。

### MCP Server

QMD 内置 MCP Server，支持 stdio 和 HTTP 两种传输方式。Agent 通过标准的 MCP 协议就能调用 QMD 的搜索能力，不需要额外的胶水代码。stdio 模式适合本地单 Agent，HTTP 模式可以多 Agent 共享同一个搜索服务。

### Context 机制

可以给集合和路径添加上下文描述。比如给 `docs/api/` 路径添加上下文："这个目录存放所有 REST API 文档，按版本号组织"。搜索结果会带回这些 context 信息，帮 Agent 判断搜到的内容属于什么领域、应该怎么使用。

这个设计的价值在于：Agent 拿到搜索结果后不只看到原文，还看到了"这段内容的定位和用途"——等于给搜索结果加了一层元信息。

### SDK 模式

除了通过 MCP 或命令行调用，QMD 还可以作为 SDK 直接嵌入 Node.js/Bun 应用。这意味着可以在代码里直接调用搜索 API，不需要启动一个独立进程。对嵌入式场景（比如把搜索能力集成到一个 Electron 应用里）很友好。

### 输出格式

`--json`、`--files`、`--xml`、`--md` 四种输出格式，天然适配不同的 Agent 消费方式。JSON 适合程序解析，Markdown 适合直接喂给 LLM，XML 适合需要结构化标签的场景，`--files` 只返回文件路径适合后续需要读取完整文件的工作流。

## 适用场景与边界

| 场景 | 适不适合用 QMD | 原因 |
|-----|--------------|------|
| 个人知识库搜索 | 适合 | 核心设计目标，Markdown 优先 |
| Agent 工作流中的文档检索 | 适合 | MCP + SDK + Context 机制，为 Agent 设计 |
| 团队共享文档搜索 | 看情况 | HTTP 模式可共享，但没有权限管理 |
| 大规模生产 RAG 系统 | 不适合 | 本地运行，不是分布式架构，扛不住高并发 |
| 非 Markdown 内容 | 有限支持 | 核心为 Markdown，代码文件有 AST 支持，但 PDF、Word 等格式不在覆盖范围内 |

一个容易踩的坑：看到 QMD 的混合搜索管线设计得很精巧，就想把它用在生产级 RAG 系统里。但 QMD 的所有模型（300M 的嵌入模型、600M 的重排序模型、1.7B 的查询扩展模型）都是按本地单机运行设计的，没有做分布式推理、没有做模型服务化、没有做多租户隔离。它的价值不在于"拿来直接用在生产系统里"，而在于"它的管线设计思路可以参考"。

## 管线设计的参考价值

QMD 的工程价值不只是"做了一个搜索引擎"，而是用一个完整的实现演示了一套混合搜索管线的设计思路。

对做 RAG 的人来说，至少有三点可以直接参考：

**第一，多路召回 + 分数融合是比单路向量搜索更稳的方案。** BM25 + 向量搜索的组合在学术界早有定论，但工程落地时"怎么融合分数"是关键问题。QMD 用 RRF 解决了跨模态分数不可比的问题，这个选择值得借鉴。

**第二，重排序不能直接替代初排。** 很多人的直觉是"有了 Reranker 就不需要精心设计初排了"，QMD 的位置感知混合说明实际情况更复杂——Reranker 和初排各有盲区，工程上需要按位置做加权混合，而不是让 Reranker 一锤定音。

**第三，分块质量决定搜索质量的上限。** 搜索引擎做得再好，分块切坏了也白搭。QMD 的结构评分分块 + 代码围栏保护 + AST 感知分块，比"按 token 数均匀切"精细得多。这一层的投入往往被低估，但它决定了搜索质量的天花板。
