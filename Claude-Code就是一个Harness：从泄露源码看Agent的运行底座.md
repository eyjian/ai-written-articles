# Claude Code 就是一个 Harness：从泄露的 51 万行源码看 Agent 的运行底座

> Claude Code 不只是一个命令行 AI 编程工具。从泄露的源码结构看，它是 Harness Engineering 在编码 Agent 领域的一个标杆实现——工具系统、权限沙箱、上下文管理、反馈回路、可观测性，五大要素全部落了地。理解 Claude Code 的架构，就是在理解一个工业级 Harness 到底长什么样。

2026 年 3 月 31 日，Anthropic 的 Claude Code v2.1.88 因缺失 `.npmignore` 配置，一个 59.8MB 的 source map 文件（`cli.js.map`）被意外包含在 npm 包中，约 51.2 万行 TypeScript 源码（1,906 个文件）由此可被完整还原。底层原因与 Bun 构建器默认生成 source map 的行为有关。这次泄露让整个 AI 工程社区第一次看到了一个成熟编码 Agent 的完整内部结构。

看完源码结构的人几乎得出了同一个判断：**Claude Code 的主体不是模型调用，而是包裹在模型外面的那套 Harness。**

模型（Claude）负责推理和决策。但让它能在真实项目里稳定写代码、改文件、跑测试、处理错误的，是外围的运行控制系统——工具、权限、上下文、反馈、验证，全部由 Harness 层管。

用一个公式概括：**Agent = Model + Harness。** Claude Code 就是这个公式的工业级实现。

## Claude Code 的 Harness 覆盖了什么

从源码结构看，Claude Code 完整实现了 Harness Engineering 的五大核心要素。其中，**工具系统的受控暴露**和**权限安全的渐进式设计**是它做得最重的两部分，值得细看。逐一拆解。

### 1. 工具系统（Tools）

Agent 光会"想"不行，还得能"动手"。Claude Code 给模型配备了一整套工具链：

- **文件操作**：读、写、编辑、搜索、glob 匹配
- **命令执行**：bash/shell，直接在项目目录里跑命令
- **代码检索**：grep、语义搜索，在大型代码库里快速定位
- **扩展能力**：Git 操作、浏览器访问、MCP 协议接入外部工具

关键点不在于工具多，而在于**工具的暴露是受控的**——不是把所有工具一股脑全给模型，而是根据当前任务阶段和权限，动态决定模型能用哪些工具。这本身就是 Harness 的约束机制在起作用。

### 2. 权限与安全（Permissions/Safety）

Claude Code 在权限控制上做得很重：

- **渐进式五级权限**：从 ReadOnly（只读）到 WorkspaceWrite（工作区写入），再到 DangerFullAccess（危险完全访问），逐级放开能力。模型默认拿到的是最小权限，只有在明确需要时才提升
- **沙箱隔离**：危险操作在受限环境中执行，子 Agent 不能生成新的子 Agent（防止递归失控），且子 Agent 没有用户交互接口——越界即被阻止
- **白名单/黑名单**：明确哪些文件和目录可以碰、哪些绝对不能碰
- **人工审批钩子**：高风险操作（如删除文件、修改配置）触发人工确认
- **全链路审计日志**：每次操作都有记录，可追溯

这不是 Prompt 里写一句"请不要删除重要文件"能解决的。Claude Code 把权限约束写在了代码层——模型根本拿不到越权的工具调用能力。

### 3. 上下文与记忆（Context/Memory）

Claude Code 的上下文管理是它区别于普通 AI 助手的核心能力之一：

- **动态上下文注入**：每次启动时自动加载项目结构、Git 状态、已有的 `CLAUDE.md` 规则
- **历史压缩**：长会话中积极压缩已完成的步骤，保留关键决策和状态
- **`CLAUDE.md` 长期记忆**：跨会话持久化的项目规则、偏好和经验教训
- **跨会话状态保持**：上一次改了什么、测试结果如何，下次打开还能接上

`CLAUDE.md` 本质上就是 Harness Engineering 里 `AGENTS.md` 的 Anthropic 实现——给 Agent 看的项目说明书，持久化存储，每次启动时加载。

### 4. 执行循环与反馈回路（Core Loop）

Claude Code 的核心执行流程是一个带反馈的闭环：

```
思考 → 选择工具 → 执行 → 观察结果 → 判断是否成功
  ↓ 成功 → 下一步
  ↓ 失败 → 分析错误 → 修正 → 重新执行
```

这不是"调一次模型就结束"。它是一个持续循环的 Agent Loop——每一步执行完都有验证，失败了能自动修正，同一个错误重复出现能打断换路径，多轮修复无果能降级处理。

从源码看，Claude Code 还内置了任务分解和依赖管理——复杂任务会被拆成子任务，按依赖关系编排执行，而不是一口气全塞给模型。

### 5. 验证与可观测性

Claude Code 的验证机制贯穿整个执行过程：

- **diff 检查**：每次文件修改后自动检查改动范围
- **测试自动运行**：改了代码自动跑相关测试，不通过就打回
- **错误捕获与反馈**：命令执行报错时，错误信息直接反馈给模型
- **状态回滚**：如果一连串修改导致项目状态恶化，能回退到之前的状态

每一步的输入、输出、工具调用、模型决策都有记录，出了问题能快速定位是哪一步、哪次调用偏了。

## Claude Code vs 传统代码补全模式

下面这张表用来对比"有完整 Harness"和"没有 Harness"到底差在哪。对比对象是传统的代码补全模式（如早期 Copilot 的行内补全），而非所有 AI 编程工具的最新形态。

| 维度 | 传统代码补全 | Claude Code |
|------|-------------|-------------|
| **执行模式** | 模型直接输出代码片段 | 模型做规划 → Harness 执行 → 验证 → 迭代 |
| **工具调用** | 无（只输出文本） | 完整工具链：文件、命令、搜索、Git |
| **权限控制** | 无 | 渐进式五级权限 + 白名单 + 人工审批 |
| **出错处理** | 用户手动修 | 自动检测 → 反馈 → 修正 → 重试 |
| **上下文** | 当前文件 + 少量上下文 | 项目全局 + 历史压缩 + 跨会话记忆 |
| **验证** | 无 | 自动测试 + diff 检查 + 状态回滚 |

> **注**：GitHub Copilot 在 2025 年底开始预览、2026 年初正式发布了 Agent Mode，已支持工具调用、文件编辑、终端执行等能力，正在向 Harness 化的方向演进。此处对比的是传统补全模式，旨在说明 Harness 有无的结构性差异，而非贬低某个具体产品。

传统补全是"模型直出"——没有 Harness。Claude Code 是"模型 + Harness"——模型负责想，Harness 负责执行、控制、验证和兜底。

这也是为什么同样用强模型，传统补全只能做行内代码续写，Claude Code 能做完整的跨文件重构、多步骤开发任务——差距不在模型，在 Harness。

## 硬约束 vs 软约束：Claude Code 的两层设计

Claude Code 的约束体系有一个值得注意的设计：**硬约束和软约束分层**。

**硬约束（代码级）**：写在 TypeScript 源码里，模型绕不过去。比如权限白名单、工具暴露控制、沙箱隔离——这些不是"建议"，是"门禁"。

**软约束（`CLAUDE.md` 规则）**：写在 Markdown 文件里，模型启动时加载到上下文。比如"这个项目用 TypeScript 严格模式""不要用 CSS Modules"——这些是规则引导，模型大概率会遵守，但技术上不是强制的。

这两层配合的效果是：高风险操作靠硬约束拦死，日常开发规范靠软约束引导。既不会管得太死导致模型什么都做不了，也不会太松导致模型乱来。

## 对做 Agent 工程的启发

Claude Code 的架构给了一个清晰的参考：**如果要做一个能在真实项目里干活的编码 Agent，Harness 层的工作量远大于模型调用本身。**

从 51 万行源码的核心功能模块看，模型调用相关的代码只是一小部分，工具系统、权限控制、上下文管理、执行循环、验证机制占了大头。当然，51 万行中也包含大量 UI、配置、兼容性处理等支撑代码（Rust 重写版仅约 2 万行核心逻辑即可印证这一点），但 Harness 层的工程量显著高于模型接入层，这个判断是站得住的。

三条可以直接借鉴的设计原则：

1. **工具暴露受控**：不是把所有能力都给模型，而是根据阶段和权限动态控制
2. **硬约束和软约束分层**：高风险靠代码拦，日常规范靠规则文件引导
3. **执行循环带验证**：每一步都有反馈，不是"调一次模型就结束"

## 结论

一句话收尾：**Claude Code 的核心竞争力不在 Claude 模型本身，而在包裹模型的那套 Harness——工具、权限、上下文、反馈、验证，五大要素缺一不可。**

理解了这一点，再看 AI 编程工具的竞争就会清楚很多：模型能力大家都在追赶，真正拉开差距的是 Harness 的工程质量。谁的 Harness 更完整、更严密、更能兜住 Agent 的不确定性，谁的产品就更能在真实项目里站住脚。
