# Harness Engineering 不是工作流：一个管编排，一个管兜底

> 很多人把 Harness Engineering 等同于工作流，觉得无非是给 AI Agent 安排任务、走流程。这个误解差得太远。工作流管的是"按什么顺序做"，Harness Engineering 管的是"怎么让 Agent 做得住、不翻车、出了问题能兜住"。两者不在一个层面上。

最近和不少人聊 AI Agent 相关的技术，发现一个特别普遍的误区：把 Harness Engineering 直接当成工作流，觉得二者就是一回事。

差别很大。工作流是一张任务清单——先做 A，再做 B，遇到情况 X 走分支 1，管的是步骤和顺序。Harness Engineering 是给 Agent 搭的一整套运行控制系统——约束、沙箱、反馈、监控、纠错全包在里面。工作流是其中的一个零件，不是全貌。

## 先给判断

两句话分清：

- **工作流是任务编排：** 定义步骤、顺序、分支和触发条件，解决"做什么、按什么顺序做"。
- **Harness Engineering 是运行控制系统：** 约束行为、隔离环境、管理上下文、闭环反馈、全程可观测，解决"怎么让 Agent 跑得住、不跑偏、出了问题能自动纠正"。

类比：工作流是流水线上的工序表，Harness Engineering 是流水线本身——包括传送带、质检站、安全围栏、监控大屏和应急开关。工序表只管"先钻孔再打磨再喷漆"，流水线管"钻孔的时候不能伤到操作员、打磨出了废品能自动挑出来、整条线出故障能紧急停机"。

## 工作流到底管什么

工作流本质上是**确定性的、预定义的任务编排**。它的核心能力是：

- **定义步骤和顺序**：A → B → C，或者 A → 如果条件 X → B1，否则 → B2
- **触发和流转**：上一步完成自动进入下一步，或者等待外部事件触发
- **分支和并行**：遇到不同条件走不同路径，某些步骤可以同时执行

传统工作流的典型场景是企业审批：员工提交申请 → 部门主管审批 → 财务核对 → 打款。每一步有明确的触发条件和下一步动作，没有意外，按部就班。

在 AI Agent 场景里，工作流同样管编排：先让 Agent 调研，再让 Agent 出方案，然后让 Agent 写代码，最后让 Agent 跑测试。步骤是固定的，顺序是预设的。

工作流管的事可以用一句话概括：**让步骤能串起来。**

但它不管这些事：

- 某一步跑偏了怎么办？
- Agent 在执行过程中"偷工减料"（声称完成但实际没做）怎么发现？
- 上下文越来越长，Agent 开始忘事怎么处理？
- Agent 访问了不该访问的文件或接口怎么拦截？
- 同一个错误反复出现怎么打断？
- 出了问题怎么定位到底是哪一步、哪次调用出的？

这些，都是 Harness Engineering 管的事。

## Harness Engineering 到底管什么

Harness Engineering 的范围比工作流大得多。这个概念在 2026 年初由 Mitchell Hashimoto 在博客中提出，随后 OpenAI、Anthropic 等团队在工程实践中进一步发展。它覆盖五个核心要素，工作流只是其中"编排"那一块的一小部分。

### 1. 约束机制——定规矩

给 Agent 设硬边界：什么能做、什么不能做。不是写在 Prompt 里的建议，而是写在执行层代码里的门禁。

- 没计划就不允许进入执行阶段
- 改了业务代码必须先跑测试
- 某些目录禁止写入
- 未通过验证就不能宣称完成

Prompt 里的 Rules 是软约束——模型可能忘、可能忽略。Harness 的约束是硬的——Agent 绕不过去。

### 2. 沙箱隔离——圈地盘

Agent 的代码执行和文件操作跑在隔离环境里（比如 Docker 容器），不直接接触宿主机和生产环境。

- Agent 执行的代码不会影响宿主机
- 每个任务有独立的文件系统
- 会话之间互不干扰
- 出了问题不会扩散

工作流不管执行环境，只管步骤顺序。Harness 管的是"Agent 在哪里干活、干活的时候不能碰什么"。

### 3. 上下文管理——管信息

Agent 在长任务中最容易出的问题是上下文越来越脏、越来越长，最后忘了重点。Harness 的上下文管理包括：

- 多 Agent 之间上下文隔离，互不污染
- 长会话积极压缩，转存中间结果
- 关键状态写到外部存储，不依赖上下文回忆
- 跨会话的长期记忆持久化

这些能力在传统工作流里完全没有对应物——工作流根本不关心信息层面的事。

### 4. 反馈回路——能纠错

Agent 执行失败时，系统能自动检测、反馈、重试，不需要人介入。

- 代码执行报错 → 错误信息自动反馈给 Agent → Agent 修正后重试
- 测试不通过 → 打回重做，直到通过
- 同一个错误重复出现 → 自动打断，强制换路径
- 多轮修复无果 → 升级到人工介入

工作流只关心步骤能不能串起来，串起来之后某一步崩了，它没有任何修复机制。

### 5. 可观测性——看得见

Agent 的每次调用、每个决策、每次工具执行都有迹可查。

- 完整的调用链路追踪
- 每次 LLM 调用的输入输出
- 工具执行结果
- Agent 的决策过程

出了问题不用靠猜，直接在追踪面板里找到"是哪个 Agent、哪一步、哪次调用偏了"。这是工作流完全覆盖不到的层面。

## 一张表看清区别

| 维度 | 工作流 | Harness Engineering |
|------|--------|-------------------|
| **核心目标** | 让步骤按顺序串起来 | 让 Agent 跑得住、不翻车 |
| **管什么** | 步骤、顺序、分支、触发条件 | 约束、沙箱、上下文、反馈、可观测 |
| **执行环境** | 不管 | 沙箱隔离，独立文件系统 |
| **出错处理** | 不管（或简单重试） | 自动检测、反馈、纠正、降级、人工升级 |
| **上下文** | 不管 | 隔离、压缩、转存、持久化 |
| **约束** | 流程层面的顺序约束 | 行为层面的硬门禁 |
| **可观测** | 流程状态（到了哪一步） | 全链路追踪（每次调用的输入输出） |
| **适用范围** | 已知、固定、确定性的任务 | 自主决策、不确定性高、长周期的 Agent 任务 |

工作流是 Harness Engineering 里的一个组成部分——大致对应"编排"这一块。不过 Harness 内的编排层（如 LangGraph 的状态机）比传统工作流更复杂，支持状态回退、条件分支和动态节点。传统工作流的核心思路——步骤和顺序——只是 Harness 编排能力的基础部分。而 Harness 还管约束、沙箱、上下文、反馈和可观测，这些工作流都不涉及。

## 一个场景看透差距

假设让 Agent 自动处理客户售后问题。

**只有工作流的版本：**

```
收到客户问题 → 分类问题 → 匹配知识库答案 → 回复客户
```

步骤清楚，能跑。但问题也很明显：

- 客户问题超出预设分类，Agent 就卡壳或乱回复
- Agent 回复了违规内容，没人发现
- 同一个错误回复反复出现，没有纠正机制
- 回复质量下降了，没有监控手段

**有 Harness 的版本：**

同样包含上面的工作流，但还多了这些：

- **约束**：回复内容必须通过合规检查，违规内容直接拦截
- **反馈**：实时监控回复准确率，发现错误自动标记，反馈给 Agent 修正
- **降级**：遇到复杂问题自动转接人工客服，而不是硬撑
- **可观测**：每次回复都有完整记录，可以回溯分析哪些问题处理得好、哪些需要改进
- **上下文管理**：长对话中保持客户历史信息，不会"上一句刚说过的事下一句就忘了"

两个版本的差距不是"多加了几条规则"，而是有没有一套系统在兜底。

## 结论

一句话收尾：**工作流管的是"步骤怎么串"，Harness Engineering 管的是"Agent 怎么稳"。一个是编排，一个是兜底。前者是后者的一个零件，不是全貌。**

把 Harness Engineering 理解成工作流，最大的风险是：搭完工作流就以为万事大吉，结果 Agent 一上长任务就跑偏、一碰复杂场景就翻车，而系统里没有任何机制能兜住。先分清这两个概念，才能搭出真正跑得住的 Agent 系统。
