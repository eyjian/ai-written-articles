# Agent Team 管分工，Harness 才是底座：两者怎么组合才对

> Agent Team 和 Harness Engineering 不是二选一。Agent Team 负责按角色把分工立起来，Harness 负责让这套协作跑得住。一个是车间班组，一个是流水线加质检加安全系统。少了后者，班组再齐整也是在手工作坊里干活。

很多人一开始接触 AI 原生研发，会先问两个问题：

第一个是"我该用 Agent Team 吗？"——答案通常是"可以"。第二个是"Harness Engineering 和 Agent Team 什么关系？"——这个问题问对了，但很多回答没说到点上。

最常见的误解是把两者放到同一个层面去比较，好像你得在"用 Agent Team"和"搞 Harness Engineering"之间选一个。不是这样。它们根本不在一层。

## 先给判断

赶时间就记这两句：

- **Agent Team 是协作组织方式：** 按角色分工，产品、架构、前端、后端、测试各管一摊，协作串起来往下跑。
- **Harness Engineering 是可控运行系统：** 给这套团队搭约束、上下文管理、反馈机制、可观测性和沙箱环境，让 Agent 稳定地自主运行。

类比也很直白：Agent Team 是车间班组，Harness 是流水线 + 质检 + 安全系统。

**没有 Harness 的 Agent Team，是一群各自很能干的工人在手工作坊里凑活儿。** 它能出东西，但你不敢不盯。

## 为什么 Harness 是底座而不是附加项

很多人把 Harness 理解成"事后加的保护层"：先拆好 Agent Team，发现跑偏了，再补几条约束。

这个理解倒着了。Harness 更应该是你先设计的那层。原因有四个。

### 1. 它解决的是 Agent 的不可控问题

Agent Team 里的每个角色都能干活，但它们没有人类团队里那种"默认的纪律"。开发 Agent 不会自己判断"这个接口我不该改"，测试 Agent 不会自动决定"这条边界用例不能省"。

Harness 的约束和门禁，就是把这种纪律变成硬规则。不满足约束就不让过，不需要人盯着。

有人会说："我给每个 Agent 的 Prompt 里加了详细的 Rules，也能约束行为啊。" 能，但靠不住。Rules 写在 Prompt 里是软约束——模型会忘、会忽略、会在上下文变长后悄悄跳过。Harness 的约束是硬约束——写在执行层代码里，Agent 绕不过去。区别不在于"有没有规则"，而在于"规则是建议还是门禁"。建议可以无视，门禁过不去就是过不去。

### 2. 它解决的是长周期任务的可靠性

短任务问题不大。写一个函数、做一个小页面，单 Agent 也能搞定。

但如果是一个跑几天甚至几周的完整研发项目，中间会有几十上百次 Agent 调用。每一次调用都有跑偏的可能，累积下来偏差会越来越大。

Harness 的反馈回路和自动化测试，就是在每一步都拉回来。不是等到最后才发现偏了。

### 3. 它解决的是多 Agent 之间的协同问题

Agent Team 最容易出事的地方不是"某个 Agent 不行"，而是"Agent 之间对不上"。

前端 Agent 按第一版接口写，后端 Agent 已经按第二版改了协议。测试 Agent 拿到的数据模型是第三版。谁都没错，但加在一起就全是 Bug。

Harness 的契约层和版本对齐机制，管的就是这件事：**统一口径，锁死协议，谁改了都得通知到位。**

### 4. 它适配"工厂化生产"的诉求

如果你追求的是"黑盒化、流水线化、少人干预"——这正好就是 Harness 要做的事。

Workflow 只能保证步骤能串起来。Harness 才能保证整条链路跑完以后，你不用再逐个检查每一步的输出。

## 最佳组合长什么样

把 Agent Team 和 Harness 放在一起看，架构上分三层：

```text
┌──────────────────────────────────────────────────┐
│           顶层 Harness（全局约束与门禁）             │
│  接口契约 │ 上下文规范 │ 部署配置 │ 全局质量门禁     │
├──────────────────────────────────────────────────┤
│           Agent Team（角色协作层）                   │
│  Product → Architect → Frontend / Backend → Test  │
│  每个 Agent 带轻量 Harness（自己的规则与校验）       │
├──────────────────────────────────────────────────┤
│           反馈闭环（自动纠偏层）                      │
│  测试失败 → 回灌对应 Agent → 自动迭代 → 复检        │
└──────────────────────────────────────────────────┘
```

几个要点值得展开。

**顶层 Harness 管的是全局。** 接口协议定了就不能随便改。上下文规范（字段命名、目录结构、代码风格）在全团队统一。部署配置和环境变量有固定来源。这些不是 Agent 自己能决定的事。

**每个 Agent 还可以带自己的轻量 Harness。** 比如 Frontend Agent 有自己的组件命名规则，Backend Agent 有自己的 SQL 安全校验，Test Agent 有自己的覆盖率门槛。这些不需要全局统一，但在各自的环节里必须管住。

**反馈闭环是把整条线拉活的关键。** 测试 Agent 发现某个接口返回 500，不需要人介入——它直接把失败报告回灌给 Backend Agent，后者修完再跑测试，直到通过。这条回路没有 Harness 是搭不出来的。

## 一张表看清两者的分工

| 维度 | Agent Team 负责 | Harness 负责 |
|------|----------------|-------------|
| **角色分工** | 谁做需求、谁做开发、谁做测试 | 不管，它不关心谁是谁 |
| **执行顺序** | 先需求再设计再开发再测试 | 不管编排，只管每一步的准入和准出 |
| **接口一致性** | 希望大家对得上 | 不按契约就编译不过 |
| **代码质量** | 各 Agent 尽力写好 | 不合规不让合并 |
| **测试回归** | Test Agent 跑测试 | 不通过就自动回滚或重跑 |
| **版本对齐** | 靠 Manager Agent 协调 | 靠契约层强制锁定 |
| **长期稳定性** | 靠分工和编排不变 | 靠反馈回路持续纠偏 |

最后一行很关键。Agent Team 的稳定性靠的是"分工和编排不变"——如果需求稳定、Agent 没抽风，结果就还行。但 Harness 的稳定性靠的是"偏了就拉回"——不管 Agent 怎么抽风，只要约束和反馈在，就不会偏到离谱。

## 落地建议：先搭 Harness，再上 Agent Team

准备上手的话，顺序很重要。别反过来。

### 第一步：搭最小 Harness

先把四件事定下来，不需要很重：

1. **上下文规范：** 字段命名、目录结构、代码风格——写在一份共享配置里，所有 Agent 启动时加载。
2. **接口契约：** 前后端之间、Agent 之间的接口协议，一旦定了就锁死。改的话必须走统一变更流程。
3. **沙箱环境：** Agent 的代码生成和测试跑在隔离环境里。不直接碰生产。
4. **自动化测试：** 至少有一层基本测试在每一步之后自动跑。不通过就不往下走。

### 第二步：上 Agent Team

Harness 搭好了，再按角色拆 Agent。拆起来会轻松很多——约束已经在了，Agent 不需要自己"猜"该遵守什么规则，Harness 会管住它。

### 第三步：强化可观测性

前两步跑通以后，补上日志、监控和回溯能力。不是为了好看，而是为了**出了问题能定位到底是哪个 Agent、哪一步、哪次调用偏了**。

没有可观测性，Harness 也会变成黑盒——你知道"系统拦住了什么"，但不知道"为什么会被拦住"。

## 结论

一句话收尾：**Agent Team 让分工立得住，Harness 让系统跑得稳。**

两者不是二选一。Agent Team 是协作组织，Harness 是底座。你要的"工厂化流水线"，必须靠 Harness 才成立。先搭底座、再上协作、最后补可观测性——这个顺序别反。
