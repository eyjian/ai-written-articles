# 没有 Harness 的 Agent Team，只是看起来像流水线

> 很多人搭 Agent Team 的第一反应是先拆工作流：需求 → 设计 → 开发 → 测试。流程拆得很漂亮，跑起来却总是偏。偏需求、偏接口、偏质量、偏上下文。问题不在流程不够全，而在流程上没有约束。

有一个类比可以把这件事说得很直白。

**Agent Team 和 Workflow**，更像是车、轮子、发动机。它们负责"能跑"。但光有车不够。你还需要方向盘、刹车、车道线和电子稳定系统。

这些东西，就是 **Harness**。

没有 Harness 的 Agent Team，本质上是一辆没有方向控制的车。它能动，但你不敢放手。

## 先给判断

如果你赶时间，就记这两句：

- **只有 Workflow 的 Agent Team = 玩具级、demo 级。** 流程能跑，但结果不可靠，人必须全程盯着。
- **有 Harness + Workflow 的 Agent Team = 生产级、可黑盒化。** 不满足约束就不让进入下一阶段，人可以真正放手。

这两者之间的差距，不是"加几条规则"就能补上的。它是两种完全不同的设计起点。

## 传统 Agent Team 到底缺什么

大多数人搭 Agent Team 的时候，想的是工作流。

先做需求分析，再做方案设计，再分前端后端各自开发，最后来一轮测试。听着很完整。

可仔细看，这些步骤只定义了**该做什么、先做什么后做什么**。

它没有定义三件更关键的事：

1. **什么不能做。**
2. **必须遵守什么。**
3. **错了怎么自动拉回来。**

结果就是：**流程能跑，但结果不可靠。** 需求 Agent 可能偷偷扩了范围，开发 Agent 可能随手改了接口字段，测试 Agent 可能跑了用例但没验边界。

你越不盯，它越飘。

这不是流程拆得不够细的问题，而是从一开始就没有在流程上加硬约束。

## Harness 到底在管什么

Harness 不是"建议 Agent 这样做"。

Harness 是**"不满足约束就不让进入下一阶段"**。

它和 Workflow 的区别，就像"开车建议走直线"和"车道线 + 电子稳定系统 + 超速断油"之间的区别。前者是建议，后者是约束。

在一条 AI 研发流水线里，Harness 至少要管住四层：

### 1. 需求边界 Harness

- 不许做需求范围外的功能
- 必须按验收标准写实现
- 不达标，直接打回需求 Agent

这一层管的是"不能多做、不能少做、不能偏着做"。

### 2. 架构 / 契约 Harness

- 接口字段、命名规范、通信协议必须固定
- 前后端只能按契约开发
- 不按契约？编译和构建直接拦死

这一层管的是"对不上号就别往下走"。

很多 Agent Team 跑偏的第一个地方就在这里：接口没锁死，前端和后端各写各的，到了联调才发现字段对不上。

### 3. 代码规范 Harness

- 格式、安全、复杂度全部有门禁
- 不合规的代码不让合并、不让部署

这一层管的是"写出来的东西必须能维护"。

### 4. 测试 Harness

- 用例不通过，自动回滚或重跑
- 不用人判断，机器直接卡控

这一层管的是"别用人肉验证来兜底"。

这四层加在一起，才构成一条有约束力的流水线。少了任何一层，流程看着再完美，也是"方向盘虚位巨大"——稍微复杂一点就偏。

## 一张表看清差别

| 维度 | 只有 Workflow | Workflow + Harness |
|------|-------------|-------------------|
| **定义了什么** | 该做什么、先做什么后做什么 | 还定义了什么不能做、必须遵守什么、错了怎么拉回 |
| **对跑偏的态度** | 靠人盯 | 靠约束自动卡控 |
| **接口一致性** | 希望对得上 | 不按契约就编译不过 |
| **代码质量** | 看运气 | 不合规不让合并 |
| **测试通过** | 跑了就算 | 不通过就回滚 |
| **能不能放手** | 不敢 | 可以 |
| **适合什么场景** | 演示、原型、探索 | 生产、交付、持续运行 |

## 为什么很多人搭 Agent Team 先想 Workflow 而不是 Harness

因为 Workflow 更直觉。

"先做 A，再做 B，再做 C"——这是人类做项目管理的自然思路。把它搬到 Agent Team 上，也很顺手。

但 Workflow 只回答了"流程怎么走"，没回答"流程走歪了怎么办"。

在人类团队里，走歪了有人盯着拉回来。Code Review 有人做，接口联调有人协调，需求跑偏有产品经理喊停。

到了 Agent Team 里，如果你还是靠人盯，那 Agent 只是替你多写了几行代码，并没有让你的流程变得更可靠。

所以更稳的做法不是"先拆 Workflow，再看要不要加约束"，而是**先想清楚 Harness 怎么设计，再把 Agent 挂上去**。

## 一个更准确的架构理解

如果你把 Harness 和 Agent Team 的关系画成一张图，会是这样：

```text
┌──────────────────────────────────────────────────┐
│                  Harness（约束层）                  │
│  需求边界 │ 契约校验 │ 代码规范 │ 测试门禁 │ 回滚  │
├──────────────────────────────────────────────────┤
│                  Workflow（编排层）                  │
│  需求 Agent → 设计 Agent → 开发 Agent → 测试 Agent │
├──────────────────────────────────────────────────┤
│                  执行层（Agent 本身）                │
│  LLM 推理 + 工具调用 + 代码生成 + 文件操作         │
└──────────────────────────────────────────────────┘
```

这张图想说的就一件事：**Agent Team 是挂在 Harness 上的执行节点，而不是反过来。**

Workflow 负责编排谁先谁后。Harness 负责定义每一步的门禁和回退条件。Agent 负责在这两层约束下干活。

把这三层的主次关系搞反了，就会出现一个很常见的问题：Agent 能力越来越强，但系统越来越不可控。

## 什么时候只有 Workflow 就够了

也不是所有场景都需要 Harness。

| 场景 | 是否需要 Harness |
|------|-----------------|
| 做个 demo，验证思路 | 通常不需要，Workflow 就够 |
| 写个一次性脚本 | 不需要 |
| 搭一条要持续运行的 AI 研发流水线 | 需要 |
| 多 Agent 协作完成一个完整交付物 | 需要 |
| 输出结果会直接进入生产环境 | 必须有 |

判断标准也不复杂：**如果你不敢不盯着它跑，那就需要 Harness。**

## 从哪里开始设计 Harness

如果你已经被说服了，下一步不是"把四层 Harness 全部搭齐"，而是先找到你的 Agent Team 最容易偏的那一层，从那里开始。

一个简单的排查顺序：

1. **先看需求。** Agent 是不是经常多做或少做？是不是经常偏离验收标准？如果是，先补需求边界 Harness。
2. **再看接口。** 前后端对不对得上？字段命名一致吗？协议有没有被改？如果对不上，先补契约 Harness。
3. **再看代码。** 输出的代码能不能直接维护？有没有安全问题？有没有复杂度爆炸？如果不行，先补代码规范 Harness。
4. **最后看测试。** 测试是不是真的在卡控？还是跑了个寂寞？如果是后者，先补测试 Harness。

不用一次全补齐。先管住最痛的那一层，就已经能把稳定性拉上一个台阶了。

## 结论

把话收回来，就一句：**不要从"工作流怎么拆"入手，要从"Harness 怎么设计"入手。**

Workflow 决定的是流程能不能跑起来。Harness 决定的是流程跑起来以后，会不会偏到你接不住。

没有 Harness 的 Agent Team，只是看起来像流水线。有 Harness 的 Agent Team，才是你真正能放手的黑盒生产线。
