# 别让 AI 裸奔：用 CodeBuddy 搭一个 Markdown 质检 Harness

这篇不聊概念，直接上手——用 `CodeBuddy` 给 AI 套一条最小的缰绳。场景也故意收得很小：就在一个本地仓库里，把 Harness 的四个核心动作（约束 → 执行 → 反馈 → 控制）跑一遍完整闭环。

> 不用外部云平台，不搞复杂多 Agent。只用 `CodeBuddy` 当前公开可见的能力，先做一条最小闭环：先约束，再执行，再复检，必要时还能回退。

很多人一听 `Harness Engineering`，就会下意识把它想成一套很重的 Agent 工程：得有平台，得有调度，还得接一堆外部系统。

第一次练，其实没必要把摊子铺这么大。

找一个本地、结果好验证、出错能回退的小任务，就够了。

这篇就拿一个具体例子来跑：**本地 Markdown 质检守门员**。它只做六类硬检查，第一轮只出报告，不改正文；第二轮只开低风险修复；修完必须复检；如果结果不对，还能退回去。

这已经是一条像样的 harness 了。

先把话说准：**`CodeBuddy` 官方并没有把这些能力整体命名成 `Harness Engineering`。** 本文讲的是——基于官方文档已经公开写明的 `Plan Mode`、`Rules`、检查点、预览/环境检查等能力，怎样把 `Harness` 这套做法落到一个简单、能复现的本地案例里。

文中涉及的能力，主要参考这几份公开资料（其中 `Rules` 的公开入口当前仍在腾讯云文档）：

- [概览页](https://www.codebuddy.ai/docs/zh/ide/User-guide/Overview)
- [Plan Mode 文档](https://www.codebuddy.ai/docs/zh/ide/Features/Plan-Mode)
- 腾讯云官方的 [Rules 文档](https://cloud.tencent.com/document/product/1749/118987)

## 先把结论说白

如果赶时间，可以先记这四句：

- **`Harness Engineering` 不是让 AI 更会写代码，而是让它更不容易乱写。**
- **在 `CodeBuddy` 里，已经可以组合出一条最小 harness 流程：`Rules + Plan Mode + 检查点 + 复检`。**
- **第一次上手，别做大工程。先挑一个本地、结果好验证、能先报告后修复的小任务。**
- **最值钱的一条纪律，是第一轮只看不改。**

## 第一章：这篇里的 `Harness`，到底指什么

别把词说玄了。

放到今天这个案例里，先按一个**裁剪版**的四层框架理解就够了。它不是完整的 `Harness Engineering` 地图，而是专门为这篇本地质检场景收窄后的版本：

| 这层东西在管什么 | 这篇怎么落 | 对应到 `CodeBuddy` 是什么 |
|------|-----------|-------------------------|
| **前馈约束** | 先把范围、禁区、产物格式说死 | `Rules` + `Plan Mode` 需求澄清 |
| **执行编排** | 把任务拆成扫描、修复、复检几步 | `Plan Mode` 的任务清单 |
| **反馈传感** | 不听口头总结，只看报告和复检结果 | 报告文件 + 环境检查 + 再扫描 |
| **状态控制** | 放权前先留回退点，出错能撤 | 检查点、历史记录、计划归档 |

这里借的是 Birgitta Böckeler 提到的 `Feedback Sensors` 思路：给 AI 的输出接上"测量仪表"，别只听它说"已经完成了"。

一句话就是：**先把 AI 放进规则里，再让它干活。**

这篇选"Markdown 质检守门员"做例子，也正是因为它够小、够硬、够好验。

## 第二章：为什么我选"Markdown 质检守门员"

因为它很适合拿来练第一条 harness。

不用配数据库。也不用起服务。更不用解释前后端交互。

只在本地仓库里处理 `.md` 文件。结果一眼能看懂。风险也相对可控。

像这些问题，读者一看就明白：

- 标题从 `##` 直接跳到 `####`
- 代码块 fence 没闭合
- 图片相对路径不存在
- 固定结尾声明丢了

这不是"模型觉得可能不太优雅"的那种软判断。

这是硬问题。

更关键的是，它天然适合做成这条顺序：

```text
先扫描 → 出报告 → 人看结果 → 留检查点 → 只开安全修复 → 再复检
```

这条顺序一旦跑通，就已经摸到 `Harness Engineering` 的门了。

我们先把这次案例的目标收窄。

| 项目 | 这次先做什么 | 这次先不做什么 |
|------|-------------|----------------|
| **扫描对象** | 当前仓库里的 `.md` 文件 | 网页、数据库、远程知识库 |
| **检查内容** | 6 条硬规则 | 文风优雅度、观点判断、事实核查 |
| **第一轮产物** | 结构化报告 | 自动批量修正文 |
| **第二轮产物** | 只修安全项 | 改标题、改段落、删正文 |
| **技术选型** | Python 标准库优先 | UI、部署、外部服务 |

## 第三章：开始前，先把三样东西写死

这一步别嫌麻烦。

前面把规则口径、命令契约和 `Rules` 文件写死，后面执行时才不会一边做一边改。

### 3.1 先把规则判定口径写死

这里建议就做 6 条规则。别多。

| 规则 | 这次怎么定义 |
|------|-------------|
| **标题层级跳跃** | 文档允许从 `#` 或 `##` 起步；后续标题层级最多只允许递进 1 级。`## -> ###` 合法，`## -> ####` 不合法。 |
| **代码块 fence 未闭合** | 成对的三反引号代码块必须闭合；未闭合就报错。 |
| **图片相对路径不存在** | Markdown 图片路径按当前 `.md` 文件所在目录解析；文件不存在就报错。 |
| **缺少固定结尾声明** | 忽略首尾空白后，文末必须包含固定声明文本；没有就报错。 |
| **空章节** | 一个标题到下一个同级或更高层标题之间，如果只有空白、分隔线，算空章节。 |
| **文末缺少空行** | 文件最后没有以换行符结束，算一个可修复问题。 |

固定结尾声明，建议这次就用仓库里现成这句：

```text
本文由 AI 原生生成，内容经本人构思并把控，仅代表个人观点，欢迎交流探讨。
```

为什么要先把判定口径写死？

因为如果"什么算跳级""图片路径相对谁算""空章节怎么判"这些都没说清，后面就算跑通了，也跑不稳。

### 3.2 再把 CLI 契约写死

别只让 `CodeBuddy` "做个脚本"。

还要把这个小工具的接口也定下来。这样它才像一个能复用的守门员，而不是一次性的对话产物。

我建议这次就用这 3 个命令：

| 命令 | 作用 |
|------|------|
| `python3 tools/markdown_guardian.py scan --root . --report reports/markdown-guardian-report.json` | 扫描并生成第一轮报告 |
| `python3 tools/markdown_guardian.py fix-safe --root . --report reports/markdown-guardian-report-after.json` | 只修低风险问题，再输出修后报告 |
| `python3 tools/markdown_guardian.py compare --before reports/markdown-guardian-report.json --after reports/markdown-guardian-report-after.json --out reports/markdown-guardian-compare.json` | 输出终端对比表，并落盘前后对比 JSON |

退出码也别含糊。

| 退出码 | 含义 |
|------|------|
| `0` | 扫描成功，且未发现问题 |
| `1` | 扫描成功，但发现问题 |
| `2` | 工具自身执行失败 |

还有一个细节，最好现在就定：`compare` 不只是"屏幕上看一眼"。我建议它同时做两层输出：

- 终端打印一张人能快速扫读的对比表
- 额外落盘 `reports/markdown-guardian-compare.json`，字段至少包含 `rule`、`before`、`after`、`remaining`、`action`

这样前面的 `scan`、`fix-safe`、`compare` 三个命令，才算把"给人看"和"给流程接"都说完整了。

### 3.3 最后把 `Rules` 文件写进去

根据腾讯云官方文档，项目级规则目录在项目根目录下的 `.codebuddy/rules`。

所以可以直接新建：

```text
.codebuddy/rules/markdown-guardian.md
```

内容可以直接抄下面这版。

```markdown
你正在为当前仓库构建一个本地 Markdown 质检工具，必须遵守以下规则。

## 目标
- 只处理当前项目中的 Markdown 文件
- 第一轮必须是 dry-run，只输出报告，不修改正文
- 第一版只实现 6 条规则：
  1. 标题层级跳跃
  2. 代码块 fence 未闭合
  3. 图片相对路径不存在
  4. 缺少固定结尾声明
  5. 空章节
  6. 文末缺少空行

## 规则判定
- 允许文档从 # 或 ## 起步
- 后续标题层级最多只允许 +1
- 图片路径按当前 Markdown 文件所在目录解析
- 固定结尾声明文本必须完全匹配
- 空章节指标题之后到下一个同级或更高层标题之间只有空白或分隔线

## 范围约束
- 忽略 `.codebuddy`、`.git`、`node_modules`
- 优先使用 Python 标准库
- 不引入外部 SaaS、数据库、部署或网页 UI
- 不把工具扩成通用 CMS

## 工具契约
- 需要实现三个命令：`scan`、`fix-safe`、`compare`
- `scan` 输出 `reports/markdown-guardian-report.json`
- `fix-safe` 只允许修复低风险项，并输出修后报告
- `compare` 输出终端对比表，并落盘 `reports/markdown-guardian-compare.json`
- 退出码：0=无问题，1=发现问题，2=脚本错误

## 输出要求
- 第一轮必须输出结构化 JSON 报告
- 每条问题至少包含：`file`、`rule`、`severity`、`line`、`message`、`fixable`
- `compare` 的 JSON 至少包含：`rule`、`before`、`after`、`remaining`、`action`
- 没有明确授权前，不允许修改 Markdown 正文

## 修复边界
- 第二阶段只有在我明确说"开启安全修复"后才允许执行
- 低风险项暂时只允许：
  - 补固定结尾声明
  - 补文末空行
- 禁止自动重写段落
- 禁止改标题文案
- 禁止删除正文内容

## 做事方式
- 如果需求不清楚，先追问
- 如果计划里出现范围外任务，先删掉再执行
- 每完成一个阶段，都要提示我先看结果
- 在开启安全修复前，提醒我先确认检查点可回退
```

## 第四章：规则写完后，别急着做——先验证它有没有被读进去

这是最容易被跳过的一步。

规则文件写在那儿，不代表当前这轮对话真的把它读进去了。

最稳的办法只有一个：**保存规则后，新开一个 `Plan Mode` 计划。**

然后在第一条提示词里，先让它复述边界。

可以这样说：

```text
我要在当前仓库里实现一个本地 Markdown 质检守门员。

开始之前，请先复述你读到的项目规则，至少包括：
- 第一轮必须 dry-run
- 这次只做 6 条规则
- 需要实现 scan / fix-safe / compare 三个命令
- fix-safe 只能修补固定结尾声明和文末空行
- 不允许自动改写正文段落

确认完这些，再进入需求澄清并输出计划。
```

如果它复述出来的内容里，少了下面任何一个：

- `dry-run`
- 6 条规则
- 3 个命令名
- 低风险修复边界
- 忽略目录

那就别执行。

直接让它重来。

这不是吹毛求疵，而是在检查：**前馈约束到底有没有真的接管执行。**

## 第五章：正式开做——用 `Plan Mode` 起一个小而稳的计划

确认规则已经被读进去之后，再给它完整目标。

可以直接贴这一版：

```text
我要在当前仓库里实现一个本地 Markdown 质检守门员。

目标：
- 扫描当前项目中的 .md 文件
- 第一轮只输出 JSON 报告，不修改正文
- 第二轮只允许 fix-safe 修复低风险问题
- 最后输出 before/after 对比

请基于当前项目规则，先做需求澄清，再给出计划。

我希望最后能得到：
1. `tools/markdown_guardian.py`
2. `reports/markdown-guardian-report.json`
3. `reports/markdown-guardian-report-after.json`
4. `reports/markdown-guardian-compare.json`
5. 一个能解释 scan / fix-safe / compare 用法的简短说明
```

正常情况下，`Plan Mode` 会先问一些澄清问题。

这时候别跳过。

如果它没主动问这些，就自己补：

```text
补充约束：
- 固定结尾声明文本是：本文由 AI 原生生成，内容经本人构思并把控，仅代表个人观点，欢迎交流探讨。
- 图片路径按 Markdown 文件所在目录解析
- `scan` 发现问题时返回退出码 1
- 第一轮不允许写回任何 .md 文件
- 第二轮修复前要先提醒我确认检查点可回退
```

## 第六章：在执行前，先审计划

别一看到计划出来就点执行。

先看这 6 件事有没有写进去：

| 计划里必须有的东西 | 通过标准 |
|------|----------|
| **扫描脚本** | 明确会生成 `tools/markdown_guardian.py` |
| **6 条规则** | 计划里没有偷偷扩成更多规则 |
| **JSON 报告** | 明确落盘到 `reports/` |
| **3 个命令** | `scan`、`fix-safe`、`compare` 都被写到计划里 |
| **第一轮 dry-run** | 明确第一轮不写回 `.md` |
| **简短说明** | 至少解释命令和退出码 |

如果在计划里看到这些东西，就删掉：

- 自动改写文章段落
- 网页 UI
- 外部部署
- 接知识库或数据库
- 一次塞进更多检查项

一句话：**一次 Plan 只做一个模块。**

越小，越稳。

正式执行前，再把整条链记成 6 步：

```text
1. 审计划
2. 第一轮扫描（dry-run）
3. 验报告
4. 留检查点
5. 只开 fix-safe
6. 复检 / compare
```

后面第七到第十章，其实就是按这 6 步依次展开。走到哪一步，就只放开哪一步的权限。

## 第七章：第一轮只做扫描，不给修复权

现在才开始执行。

但第一轮要盯住一句话：**只扫描，不修文。**

如果执行过程中，`CodeBuddy` 表现出要直接改 `.md` 文件，就立刻打断它：

```text
停一下。第一轮只允许生成扫描器和报告，不允许修改现有 Markdown 正文。请严格保持 dry-run。
```

第一轮跑完后，理想中应该拿到这些产物：

```text
tools/markdown_guardian.py
reports/markdown-guardian-report.json
```

报告最少要长这样：

```json
{
  "summary": {
    "files_scanned": 6,
    "issues_found": 4,
    "fixable_issues": 1,
    "non_fixable_issues": 3
  },
  "issues": [
    {
      "file": "Superpowers是什么.md",
      "rule": "missing_footer",
      "severity": "warning",
      "line": 246,
      "message": "缺少固定结尾声明",
      "fixable": true
    },
    {
      "file": "使用CodeBuddy-IDE构建Sub-Agent.md",
      "rule": "heading_level_skip",
      "severity": "error",
      "line": 88,
      "message": "标题层级从 ## 直接跳到 ####",
      "fixable": false
    },
    {
      "file": "使用CodeBuddy-IDE构建Agent-Team.md",
      "rule": "unclosed_code_fence",
      "severity": "error",
      "line": 142,
      "message": "代码块 fence 未闭合",
      "fixable": false
    },
    {
      "file": "Trae也能做Harness-Engineering吗？CodeBuddy、Trae与DeerFlow的真正区别.md",
      "rule": "empty_section",
      "severity": "warning",
      "line": 168,
      "message": "标题"### 再往下一步"下没有有效内容",
      "fixable": false
    }
  ]
}
```

如果仓库当前没有命中 `empty_section`，也很正常。这里把它放进示例，只是为了把这 6 条规则在报告里的长相一次讲全。

字段名不必一模一样。

但这几个意思必须有：

- 扫了多少文件
- 发现多少问题
- 哪些可修
- 每条问题在哪个文件、哪一行、是什么

如果报告里没有 `fixable`，或者没有明确的 `rule` 字段，那就先别往下走。

这说明反馈传感这层还没装好。

## 第八章：报告出来后，先验结果，再留检查点

到这里最容易出事：AI 口头说"检查完成"，很多人就顺手把第二轮放开了。

先别急。

先自己过一遍报告。

重点看这四件事：

| 要看什么 | 合格标准 |
|------|----------|
| **规则有没有混** | 没夹带主观评价 |
| **命中位置准不准** | 文件名、行号、问题描述基本对得上 |
| **`fixable` 标得准不准** | 只有低风险问题才标 `true` |
| **忽略目录有没有生效** | `.codebuddy`、`.git`、`node_modules` 没被扫进去 |

如果这四件事都过了，再做下一步：**先确认检查点可回退。**

这一步别省。

因为接下来要给工具一点点修改权了。

稳妥做法是：

- 如果界面支持手动创建检查点，就在这里留一个
- 如果是自动捕获编辑前状态，至少先确认当前这一步是可回退的
- 不确定时，就先停下来确认，不要直接放开 `fix-safe`

一句话：**先把退路留好，再给权限。**

如果第二轮修坏了，就回到这个点，重新收窄修复范围。

## 第九章：第二轮只开 `fix-safe`

现在才进入修复阶段。

注意，不是"把问题都修了"。

只是"只修低风险项"。

可以直接对 `CodeBuddy` 说：

```text
现在进入第二阶段，只开启 fix-safe。

要求：
- 只允许修复 `fixable=true` 的问题
- 当前只允许修复两类问题：
  1. 缺少固定结尾声明
  2. 文末缺少空行
- 不允许改动现有段落内容
- 不允许重写标题
- 不允许删除正文
- 修复后输出 `reports/markdown-guardian-report-after.json`
```

这一段最能看出有没有 harness。

同样是"让 AI 动手改文件"，差别其实很大：

| 没有 harness 的改法 | 有 harness 的改法 |
|------|----------------|
| 直接说"把问题都修了" | 只放开 `fix-safe` |
| 不区分高低风险 | 只修 `fixable=true` |
| 修完就当完成 | 修完必须再扫一遍 |
| 没留退路 | 放权前先确认检查点 |

AI 没变，变的是给它的工作制度。

## 第十章：修完必须复检，不复检就别说完成

第二轮跑完后，马上让它再扫一次，或者直接跑 `compare`。

可以补这一句：

```text
请执行 compare：终端输出 before/after 对比表，并写出 `reports/markdown-guardian-compare.json`。字段至少包含：规则名、修复前数量、修复后数量、剩余问题、是否需要人工处理。
```

这里最好约定成两层输出：

- 终端表格给人快速扫读
- JSON 文件给后续流程继续消费

理想中的表，大概长这样：

| 规则 | 修复前 | 修复后 | 剩余问题 | 处理方式 |
|------|------|------|---------|---------|
| **缺少固定结尾声明** | 2 | 0 | 0 | 已自动修复 |
| **文末缺少空行** | 3 | 0 | 0 | 已自动修复 |
| **图片路径不存在** | 1 | 1 | 1 | 人工确认路径 |
| **标题层级跳跃** | 2 | 2 | 2 | 人工判断结构 |

如果想把这一步接进后续流程，落盘 JSON 最少可以长这样：

```json
[
  {
    "rule": "missing_footer",
    "before": 2,
    "after": 0,
    "remaining": 0,
    "action": "auto_fixed"
  },
  {
    "rule": "broken_image_path",
    "before": 1,
    "after": 1,
    "remaining": 1,
    "action": "manual_review"
  }
]
```

人看表，流程接 JSON。这样 `compare` 的契约才算完整。

如果对比结果不合理，比如：

- 低风险项没降下来
- 不可修项数量莫名变少
- 第二轮顺手改了正文段落

那就别硬往前走。

直接回到上一步的检查点，重新收窄 `fix-safe` 的权限。

这才叫闭环。

## 第十一章：如果看到这些，就说明整条链跑偏了

这一章可以直接当排错表用。

| 失败迹象 | 说明哪里跑偏了 | 怎么拉回来 |
|------|----------------|-----------|
| **第一轮就改了正文** | `dry-run` 没卡住 | 停止执行，回到计划阶段，重申第一轮只扫描 |
| **报告没有 `fixable` 字段** | 反馈传感这一层不完整 | 补齐报告结构，再决定要不要进入修复 |
| **规则名和问题类型混乱** | 判定口径没写死 | 回到规则定义，补明确细则 |
| **第二轮修掉了标题或段落** | 修复边界过宽 | 回退到检查点，只保留 footer/newline 两类修复 |
| **计划里混进 UI、部署、数据库** | 范围失控 | 删掉范围外任务，重新生成小计划 |
| **扫进 `.codebuddy` 等目录** | 忽略规则没生效 | 回到 `Rules`，重发新计划并要求先复述边界 |

这张表背后的逻辑很简单：

**先确定是哪一层坏了，再修那一层。**

别一出错就整条链重做。

## 第十二章：这套做法适合谁，不适合谁

不是所有任务都值得这么做。

先分清楚。

| 场景 | 适不适合先用这套 | 原因 |
|------|----------------|------|
| **本地脚本、文档检查、代码守门** | 适合 | 规则硬，结果好验证 |
| **中等复杂的多文件改造** | 适合 | `Plan Mode` 很能发挥作用 |
| **需要长期重复执行的质量检查** | 适合 | 很适合沉淀成 `Rules` 或 `Skills` |
| **依赖大量外部系统的企业级 Agent 平台** | 不适合从这篇起步 | 变量太多，一开始容易乱 |
| **复杂多角色协作编排** | 不适合第一篇就做 | 先把单回路控制练熟再说 |

说白了，这篇适合拿来练**纪律**。

不是拿来炫规模。

## 第十三章：跑通之后，下一步该加什么

如果把这条最小闭环跑通了，后面可以按这个顺序往上加：

| 下一步 | 什么时候再加 | 作用 |
|------|-------------|------|
| **`Skills`** | 这套流程会反复用 | 把这条工作流封成固定能力 |
| **`MCP`** | 需要接外部工具或数据源 | 扩大上下文和工具边界 |
| **更细的检查点策略** | 开始做高风险批量修改 | 提高回滚安全性 |
| **接 CI 或本地提交前检查** | 想把它从"会话操作"变成"固定守门" | 让脚本真正进入研发流程 |

顺序别反。

先把一次做好，再谈规模化。

## 结论

很多人把 `Harness Engineering` 想得太重，好像非得先有一整套 Agent 平台，才谈得上控制面、反馈回路和状态治理。

第一次练，其实不必从重系统起步。

像这篇这样——先用 `Rules` 卡边界，再用 `Plan Mode` 收范围，第一轮只扫描不修文，修复前先留检查点，第二轮只开 `fix-safe`，修完再跑 `compare`——就已经是一条完整的小闭环。

如果今天就想试，建议先做五件事：

1. **先写死规则判定和命令契约**
2. **保存 `Rules` 后，新开计划验证它已经生效**
3. **第一轮只扫描，不给修复权**
4. **第二轮放权前，先确认检查点可回退**
5. **修完必须跑 `compare`，别听口头总结**

把这五步跑顺以后，会发现，AI 并没有突然更聪明；真正变化的是，它开始在一套明确制度里工作。

那时候再回头看 `Harness Engineering`，它就不再是个大词，而是一套能落到项目里的工程习惯。
