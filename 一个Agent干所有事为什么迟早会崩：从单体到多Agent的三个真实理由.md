# 一个 Agent 干所有事，为什么迟早会崩：从单体到多 Agent 的三个真实理由

> 单体 Agent 不是不能用，但任务一复杂就撑不住。角色模糊让 prompt 臃肿到失控，上下文拥挤让推理链条断裂，视角单一让错误自我放大。多 Agent 不是为了架构好看，是工程上绕不过去。

前段时间有人给我看了一份 Agent 的 system prompt——3000 多字，密密麻麻，从需求分析到架构设计到编码实现到代码审查到测试验证，全塞在一个 prompt 里。他说这个 Agent "什么都能做"。

我看了两分钟就知道问题在哪了：这个 Agent 什么都做，所以什么都做不好。

它写代码的时候会突然蹦出一段需求分析的口吻，审查代码的时候会顺手把自己写的 Bug 合理化，做测试的时候会跳过自己刚实现的边界条件。不是模型不行，而是**你让一个人同时扮演产品经理、架构师、开发、测试和审查员，他每一个角色都会打折扣**。

这不是个别现象。几乎所有从单体 Agent 起步的项目，任务一旦超过"写个小脚本"的规模，都会撞上同一堵墙。这堵墙不是模型能力不够，而是单体架构本身的三个结构性问题。

---

## 先给判断

赶时间就带走这三句：

- **角色模糊**：一个 prompt 里塞进去多种角色的指令，模型在冲突的指令之间反复摇摆，产出越来越不稳定。
- **注意力分散**：所有任务的上下文挤在同一个窗口里，推理链条越长，中间噪音越多，关键信息被淹没。
- **视角单一**：自己写的东西自己查，查不出问题。没有外部视角的校验，错误会一路放大。

这三个问题不是"优化 prompt"能解决的。它们是单体架构的结构性瓶颈，解法是把一个 Agent 拆成多个。

---

## 一、痛点 1：角色模糊——一个 prompt 塞进去三个人的活

### 什么是角色模糊

当你在一个 Agent 的 prompt 里同时写了"你是需求分析师""你是开发工程师""你是代码审查员"，模型就会面临一个根本性的困境：**我到底该以哪个身份来思考当前的任务？**

这不是假设。来看一个真实场景。

### 案例：一个 Agent 既分析需求又写代码又审查

你让单体 Agent 做一个待办事项应用。prompt 里写着：

```text
你是一个全栈开发 Agent。你需要：
1. 先分析需求，明确功能边界
2. 设计技术方案和接口
3. 编写前后端代码
4. 对自己写的代码进行审查
5. 编写测试用例并执行
```

看着很全面。但跑起来你会发现：

- **分析需求时**，它会不自觉地往"我能实现什么"上靠，而不是"用户到底需要什么"——因为"开发者"的身份在干扰"产品经理"的判断。
- **写代码时**，它可能会偷偷降低需求标准，因为它"知道"自己后面还要审查这段代码，潜意识里会把标准放松。
- **审查代码时**，它会合理化自己刚写的逻辑——"这段代码是我 5 分钟前写的，当然是对的"。

### 为什么模型会崩

LLM 的推理依赖 prompt 里的角色设定和指令。当多个角色的指令出现在同一个 prompt 里，模型的注意力会在不同指令之间分散。遇到冲突的指令（比如"尽快完成代码"和"严格审查每一行"），模型要么二选一，要么两头妥协——无论哪种，产出都不稳定。

prompt 越长、角色越多，这种冲突就越严重。3000 字的全能 prompt 不是在给模型增加能力，而是在**给它制造噪音**。

## 二、痛点 2：注意力分散——上下文窗口变成了垃圾场

### 什么是注意力分散

单体 Agent 只有一个上下文窗口。所有任务——需求分析的结论、架构设计的细节、前端代码、后端代码、测试报告——全部挤在同一个窗口里。

当模型在处理后端接口时，窗口里还残留着前端组件的变量名、需求文档的自然语言描述、上一轮对话的调试日志。这些东西对当前任务没有帮助，但它们会**占用模型的注意力预算**。用更技术的话说：上下文的信噪比急剧下降——有效信息被大量无关 token 稀释，模型的注意力机制在噪音中搜索关键信息的难度显著增加。

### 案例：任务 A 的中间结果污染了任务 B

你让单体 Agent 先设计数据库表结构，再写后端接口，再写前端页面。

- 设计表结构时，上下文干净，输出质量不错。
- 写后端接口时，上下文里多了表结构的信息，这些信息有用，但也混进了设计阶段的讨论过程（"要不要加一个字段""加了又删了"）。
- **到了写前端页面时**，上下文已经被表结构、接口代码、中间讨论、调试日志塞满了。模型在写一个简单的表单组件时，注意力被前面几千 token 的无关信息拉扯，写出来的前端代码里会莫名出现后端的命名风格、数据库的字段名，甚至上一轮对话里已经否决的设计方案。

### 为什么模型会崩

LLM 的推理质量和上下文的信噪比直接相关。上下文越长、噪音越多，模型的有效推理能力就越弱。这不是"上下文窗口不够大"的问题——即使窗口有 100 万 token，如果里面 80% 是噪音，模型的表现也不会好。

推理链条过长时，中间步骤的噪音会**累积放大**。一开始只是"稍微有点受干扰"，到第五步就变成"完全跑偏了"。

## 三、痛点 3：视角单一——自己查不出自己的错

### 什么是视角单一

让同一个 Agent 既生成内容又审查内容，就像让一个作者自己校对自己的文章——他会不自觉地跳过错误，因为大脑会自动补全它"预期看到"的内容，而不是"实际写了"的内容。

LLM 也一样。它在生成代码时建立了一套内部逻辑，审查时它会沿着同样的逻辑链条验证——结论当然是"没问题"。它不是故意放水，而是**它的视角和生成时一模一样，看不到自己的盲点**。

### 案例：Agent 生成的代码自己审查，审不出问题

你让单体 Agent 写了一个用户认证模块，然后在同一个对话里让它审查自己的代码。

它会说："代码逻辑清晰，错误处理完备，测试覆盖充分。"

但换一个独立的 Agent 来审——一个没有参与编码过程、带着全新视角的 reviewer——它可能立刻指出：

- 密码哈希用了过时的算法
- 登录失败没有做频率限制
- 异常处理只覆盖了 happy path

这些问题不是单体 Agent "看不见"，而是它**没有动力去质疑自己刚才的决策**。正应了那句话：不识庐山真面目，只缘身在此山中。

## 四、解法：多 Agent 架构怎么解这三个问题

三个痛点，对应三个解法。一张表看清楚：

| 单体痛点 | 根本原因 | 多 Agent 解法 | 怎么起作用 |
|---------|---------|--------------|-----------|
| **角色模糊** | 多种角色指令挤在一个 prompt 里 | **角色原子化** | 每个 Agent 只做一件事，prompt 短而纯粹，不会互相冲突 |
| **注意力分散** | 所有任务上下文挤在一个窗口里 | **上下文隔离** | 每个 Agent 有独立上下文，只装和自己任务相关的信息 |
| **视角单一** | 同一个模型生成又审查 | **交叉校验 + 动态编排** | 写代码的和审代码的是不同 Agent，视角天然不同 |

### 角色原子化：一个 Agent 只做一件事

把"全栈开发 Agent"拆成 5 个独立 Agent：

| Agent | 只做什么 | prompt 长度 |
|-------|---------|-----------|
| Product Agent | 需求分析 | 短 |
| Architect Agent | 架构设计 | 短 |
| Dev Agent | 编码实现 | 短 |
| Reviewer Agent | 代码审查 | 短 |
| Test Agent | 测试验证 | 短 |

每个 Agent 的 prompt 只需要定义一个角色、一套规则。不会出现"你既是开发者又是审查员"的冲突。prompt 从 3000 字压缩到几百字，模型的遵循度显著提升。

### 上下文隔离：各管各的信息

拆成多 Agent 后，每个 Agent 有自己独立的上下文窗口：

- Product Agent 的上下文里只有需求信息
- Dev Agent 的上下文里只有架构设计和接口定义
- Test Agent 的上下文里只有代码和测试规范

前端代码不会污染后端推理，需求讨论不会干扰测试执行。**每个 Agent 的上下文信噪比都很高**，推理质量自然更好。

### 交叉校验：让不同的人来查

Reviewer Agent 没有参与编码过程。它拿到代码时，建立的是全新的理解链条——这意味着它不会"顺着原来的逻辑走"，而是真正在做独立审查。

更进一步，主 Agent（协调者）可以做**动态编排**：

- 审查不通过 → 自动打回给 Dev Agent
- Dev Agent 修完 → 再次提交审查
- 审查通过 → 流转到 Test Agent

这个循环不需要人介入。协调者负责观测任务状态、路由任务、做质量校验。这就是素材里提到的"动态编排"——主 Agent 负责任务拆解与路由，任务观测，进行任务校验。

### 额外收益：LLM 混合架构

多 Agent 还带来一个单体架构做不到的工程优势：**不同 Agent 可以用不同的模型**。

Reviewer Agent 需要更强的推理能力，可以用推理更强的模型；Dev Agent 需要更好的代码生成能力，可以用代码能力更强的模型；Product Agent 主要做自然语言理解，用通用模型就够了。

单体 Agent 只能用一个模型做所有事，要么用最强的（贵），要么用便宜的（质量打折）。多 Agent 架构下，你可以**按角色选模型**，在成本和质量之间找到更好的平衡点。

## 五、一个案例跑通：从单体到多 Agent 的重构

### 场景：做一个待办事项 Web 应用

**重构前（单体 Agent）：**

```text
一个 Agent，一个 3000 字 prompt，一个上下文窗口。
从需求分析到代码实现到测试全在一个对话里完成。

结果：
- 前端代码混入后端命名风格
- 接口字段前后不一致
- 测试只覆盖了 happy path
- "代码审查"变成了自我表扬
```

**重构后（多 Agent）：**

```text
Product Agent（需求文档 + 验收标准）
    ↓
Architect Agent（接口定义 + 数据模型）
    ↓
Dev Agent（按接口契约写代码）
    ↓
Reviewer Agent（独立审查，不通过就打回）
    ↓
Test Agent（按验收标准写测试，跑测试）
    ↓
协调者全程做：任务分派、状态观测、版本对齐

结果：
- 每个 Agent 只管自己那一摊，产出更专注
- 上下文干净，不会互相污染
- 审查是真审查，不是自我安慰
- 不通过就打回，有闭环
```

关键变化不在于 Agent 的数量，而在于三件事都到位了：**角色清晰了、上下文干净了、校验独立了**。

## 六、不是所有场景都需要多 Agent

多 Agent 不是万能药。以下场景单体 Agent 就够了：

| 场景 | 建议 |
|------|------|
| 写个一次性脚本 | 单 Agent 就够 |
| 改几行配置 | 单 Agent 就够 |
| 做个快速原型验证 | 单 Agent 就够 |
| 问一个技术问题 | 单 Agent 就够 |
| 开发一个有前后端分离、有数据库、有测试的完整项目 | 该上多 Agent |
| 需要代码审查、质量门禁和自动化测试的工程项目 | 该上多 Agent |
| 任务复杂到"一个人很难不出错" | 该上多 Agent |

判断标准可以简化成一句：**如果你发现单体 Agent 开始"说着说着就跑偏了"或者"自己检查自己总说没问题"，就是该拆的信号。**

---

## 结论

单体 Agent 的三个痛点——角色模糊、注意力分散、视角单一——不是优化 prompt 能解决的，它们是架构层面的结构性瓶颈。多 Agent 用角色原子化、上下文隔离和交叉校验分别对症下药，再加上 LLM 混合架构在成本和质量上的灵活性，形成了一套工程上说得通的解法。

不是所有任务都需要拆。但当你的 Agent 开始跑偏、开始自我放水、开始"说着说着就忘了前面的事"，那不是模型的问题，是一个人干了五个人的活。该拆了。
