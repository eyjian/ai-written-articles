# Harness 的自动化验证怎么落地？CodeBuddy 与 DeerFlow 的能力边界

> Harness Engineering 里最低成本、最高收益的一层能力，就是自动化验证。但"自动化验证"不是一个开关——用不同工具实现，深度差异很大。

Harness Engineering 的三根支柱——可读性、防御机制、反馈回路——里面，反馈回路是最容易被低估的一根。

很多人把 `AGENTS.md` 写得很漂亮，状态机也画了，权限边界也列了，但最后 Agent 改完代码，既没跑测试，也没人复查，就直接宣布"任务完成"。

问题出在哪？

不是约束不够，而是**验证这一层根本没接上**。

自动化验证的核心逻辑其实很简单：

```text
Agent 写代码 → 自动跑 typecheck / lint / test → 失败则反馈给 Agent → Agent 修复 → 再验证 → 直到通过或升级人工
```

这是一个**闭环反馈机制**。关键在两件事：谁来触发验证，验证失败后怎么把结果喂回去。

### 这个闭环发生在哪里？

先别急着看具体工具。先搞清楚一件事：**上面这个闭环，不是 Agent 自己在脑子里跑的，而是有一个外部程序在驱动它。**

根据实现方式不同，这个“外部程序”可能是：

| 实现方式 | “外部程序”是谁 | 验证怎么触发 |
|---------|-------------|-------------|
| **CodeBuddy**（IDE 内的 AI 助手） | CodeBuddy 自己的对话引擎 | 你通过 Skill 规则告诉它“改完代码必须跑测试”，它在对话中自动执行 |
| **DeerFlow**（独立的 Agent 运行时） | 你自己写的 Python 编排程序 | 程序流程走到验证节点时自动触发，不通过就自动打回 |
| **自己写编排层** | 你用 Python + LLM SDK 写的管理程序 | 和 DeerFlow 类似，但所有逻辑都得自己实现 |

用一个更直观的类比：

- **CodeBuddy** 像是你给实习生写了一份工作手册：“改完代码记得跑测试”。实习生大多数时候会照做，但偶尔也会忘、会跳步。
- **DeerFlow** 像是你把这些规则写进了 CI/CD 流水线：代码提交后自动触发测试，测试不过合并按钮直接灰掉。不管谁提交，规则都一样执行。

下面分别拆开看这两种方式具体能做到什么。

---

## 一、CodeBuddy 能做到什么程度

结论先放前面：**能做到，但有边界。**

CodeBuddy 本质上是一个对话式 coding agent。它的能力边界决定了它能实现的验证方式：

| 能力 | 是否支持 | 说明 |
|------|---------|------|
| 读写文件 | ✅ | 可以修改代码 |
| 执行终端命令 | ✅ | 可以跑 `pnpm test`、`pnpm lint` 等 |
| 根据命令输出做判断 | ✅ | 看到测试失败会尝试修复 |
| Skill 定义工作流 | ✅ | 可以用 SKILL.md 约束行为 |
| 硬约束状态机 | ❌ | 没有真正的代码级状态机拦截 |
| 多 Agent 评审 | ❌ | 单会话单 Agent，无法拆成执行者 + 审查者 |

这意味着，CodeBuddy 能实现的自动化验证，更像是一种**Skill 驱动的软约束验证**——不是代码级硬约束，而是通过规则文本 + 终端命令执行来完成闭环。

### 方式一：通过 Skill 规则约束行为

最直接的做法，是写一个 Skill，要求 CodeBuddy 每次改完代码后必须执行验证命令：

```markdown
---
name: auto-verify
description: 自动验证技能。当 Agent 修改了代码文件后自动触发，强制执行测试和检查。
---

# 自动验证

## 铁律
每次修改 src/ 下的代码文件后，**必须按顺序执行以下命令**：

1. `pnpm typecheck`
2. `pnpm lint`
3. `pnpm test --run`

## 规则
- 如果任何一步失败，**必须先修复再继续**
- 禁止在验证未通过的情况下宣称任务完成
- 每轮修复后重新执行全部验证
- 连续 3 轮修复仍未通过，停下来报告问题，等待人工介入
```

**这个 Skill 文件放在哪？怎么生效？**

在 CodeBuddy 里，Skill 文件放在项目根目录的 `.codebuddy/skills/` 下，用 Markdown 编写。CodeBuddy 启动对话时会自动读取这些文件，把里面的规则当作行为约束。所以你不需要每次手动提醒它“记得跑测试”——只要 Skill 文件在，它就会读到。

但这里有一个关键区别：**Skill 是“建议”，不是“强制”。** CodeBuddy 读到规则后会尽量遵守，但它没有一个代码级的拦截器来确保“不跑测试就不允许继续”。如果上下文变长、任务变复杂，它偶尔会跳过这一步。

### 方式二：在对话中手动触发

更轻量的做法是，每次让 CodeBuddy 写完代码后，手动说一句：

> "跑一下测试，看看有没有问题。"

CodeBuddy 会执行终端命令，看到失败输出后自动尝试修复。

这种方式没有 Skill 那么系统化，但在日常开发中已经够用。

### CodeBuddy 做不到的部分

Skill 能约束行为，但它本质上是"建议"，不是"强制"。以下这些能力，CodeBuddy 目前做不到：

| Harness 文章描述的能力 | CodeBuddy 的局限 |
|----------------------|-----------------|
| **状态机锁定阶段**（RESEARCH → PLAN → EXECUTE → VERIFY） | 没有代码级的状态机拦截，Skill 只是"建议"，不是"强制" |
| **权限边界**（禁止写 `.env`、`secrets/`） | 只能在 Skill 里写"请不要碰"，但没有真正的文件系统 ACL 拦截 |
| **循环失败检测**（熔断器） | 没有内置的重试计数器，需要靠 Skill 文字约束 |
| **多 Agent 评审**（Agent A 写，Agent B 审） | 单会话单 Agent，无法拆分角色 |
| **错误经验持久化**（lessons-learned.md 自动更新） | 不会自动维护经验文件，需要手动要求 |

一句话总结：CodeBuddy 能实现 **"软约束版"的自动化验证**— —通过 Skill 规则 + 终端命令执行，做到"改完代码跑测试、失败了修复再跑"的基本闭环。但它做不到代码级硬约束（状态机、权限拦截、熔断器），因为它本质上是一个对话式 Agent，不是一个可编程的 Agent 运行时。

## 二、DeerFlow 能做到什么程度

结论也先放前面：**能做到更多，但需要你自己编排。**

DeerFlow（字节开源的多 Agent 工作流框架）的定位和 CodeBuddy 完全不同：

| 维度 | CodeBuddy | DeerFlow |
|------|-----------|----------|
| 本质 | 对话式 coding agent | 可编程的多 Agent 工作流框架 |
| 执行模型 | 单 Agent 单会话 | 多 Agent 编排，支持并行 / 串行 |
| 约束方式 | Skill 文本约束（软） | 代码级流程控制（硬） |
| 自定义程度 | 低（只能写 Skill） | 高（Python 代码控制全流程） |
| 上手成本 | 低 | 高（需要写代码搭流程） |

DeerFlow 基于 LangGraph 构建，天然支持**状态机 + 多 Agent + 条件分支**。Harness Engineering 文章里描述的几乎所有验证机制，它都能实现。

**下面的代码示例跑在哪里？**

和 CodeBuddy 不同，DeerFlow 的代码不是跑在 IDE 里的。它们跑在一个**自己写的 Python 编排程序**里——这个程序就是 Agent 的“管理员”。它负责启动 Agent、决定当前在哪个阶段、检查结果、决定是继续还是打回。下面的 `verify_node`、`verify_with_circuit_breaker` 这些函数，都是这个编排程序里的节点。

### 1. 状态机锁定阶段

DeerFlow 的工作流本身就是一个状态图，可以定义：

```text
RESEARCH → PLAN → EXECUTE → VERIFY → (通过则完成 / 失败则回到 EXECUTE)
```

每个节点只允许调用特定工具——这就是硬约束。Agent 不是"被建议"不要跳步，而是**根本跳不过去**。

### 2. 多 Agent 评审

DeerFlow 支持多个 Agent 节点，可以搭一个：

```text
Coder Agent（写代码）→ Reviewer Agent（审查）→ 条件判断 → 通过 / 打回
```

这就是 Harness Engineering 里说的"执行和评审拆开"。让写代码的 Agent 和审代码的 Agent 不是同一个，避免"自己写自己夸"。

### 3. 自动化验证回路

在 VERIFY 节点里调用终端命令跑测试，根据结果决定是通过还是打回：

```python
# 伪代码示意
def verify_node(state):
    result = run_command("pnpm test --run")
    if result.success:
        return "complete"
    else:
        state["feedback"] = result.stderr
        return "retry"  # 回到 EXECUTE 节点
```

这不是"建议 Agent 去跑测试"，而是**流程本身就会走到验证节点**，不通过就自动打回。

### 4. 循环失败检测

在状态里维护一个重试计数器，超过阈值就跳到"人工介入"节点：

```python
def verify_with_circuit_breaker(state):
    state["retry_count"] = state.get("retry_count", 0) + 1
    
    if state["retry_count"] > 3:
        return "escalate_to_human"  # 熔断，升级人工
    
    result = run_command("pnpm test --run")
    if result.success:
        return "complete"
    else:
        state["feedback"] = result.stderr
        return "retry"
```

这和传统后端里的熔断器思路一样：连续失败 → 自动打断 → 强制换路径。

### 5. 权限边界

在工具调用层加拦截逻辑，禁止写入特定路径。不是在提示词里写"别碰生产配置"，而是在代码层直接拦住。

### DeerFlow 做不到（或不方便）的部分

| 能力 | 说明 |
|------|------|
| **开箱即用** | 需要自己写 Python 代码搭建工作流，不是配置一下就能跑 |
| **IDE 集成** | 不像 CodeBuddy 那样嵌在 IDE 里，更适合跑后台任务 |
| **交互式开发** | 不适合"边聊边写"的场景，更适合"定义好流程自动跑" |

## 三、对比总结

把两者的能力摆在一起看，差别就很清楚了：

| 能力 | CodeBuddy | DeerFlow |
|------|-----------|----------|
| 改完代码自动跑测试 | ✅ 通过 Skill + 终端命令 | ✅ 通过工作流节点 |
| 测试失败自动修复 | ✅ 看到输出会尝试修复 | ✅ 状态机回到 EXECUTE |
| 状态机硬约束 | ❌ 只有软约束 | ✅ 原生支持 |
| 多 Agent 评审 | ❌ 单 Agent | ✅ 多节点多 Agent |
| 权限边界拦截 | ❌ 只能文字提醒 | ✅ 代码级拦截 |
| 循环失败熔断 | ❌ 无内置机制 | ✅ 状态计数器 |
| 错误经验持久化 | ⚠️ 需手动要求 | ✅ 可写入文件 / 数据库 |
| 上手成本 | 低（写 Skill 即可） | 高（需写 Python 编排） |
| 适合场景 | 日常交互式开发 | 自动化流水线 / 后台任务 |

这张表的核心信息是：**CodeBuddy 能做到基本闭环，DeerFlow 能做到系统级闭环。**

把这张表翻译成更直白的决策建议：

- 如果你只需要“改完代码跑个测试、失败了修一下”——CodeBuddy 已经够用，不需要上 DeerFlow
- 如果你需要“不管 Agent 想不想跑测试，系统都会强制执行”——那就需要 DeerFlow 这种硬约束
- 如果你需要“写代码的 Agent 和审代码的 Agent 不是同一个”——只有 DeerFlow 能做到
- 如果你需要“连续失败三次自动停下来”——CodeBuddy 靠 Skill 文字提醒，DeerFlow 靠代码级熔断器

差别不在“能不能跑测试”，而在约束是软的还是硬的、验证是建议的还是强制的、失败处理是靠文字提醒还是靠代码拦截。

## 四、软约束和硬约束的真正区别

这里值得多说一层。

很多人觉得"软约束也够了，反正 Agent 大多数时候会听话"。这在简单任务里确实成立。但一旦任务变复杂——上下文变长、步骤变多、工具调用变频繁——软约束的失效率会明显上升。

| 场景 | 软约束（Skill 文字规则） | 硬约束（代码级流程控制） |
|------|------------------------|------------------------|
| 简单任务（改一个文件、跑一个命令） | 够用 | 没必要 |
| 中等任务（跨文件修改、多步骤） | 大多数时候够用，偶尔跳步 | 更稳 |
| 复杂任务（长上下文、多工具、多阶段） | 容易失效 | 必须有 |
| 高风险操作（写生产配置、删数据） | 不够 | 必须有 |

这不是在说 CodeBuddy 不好。而是在说：**不同复杂度的任务，需要不同强度的约束。**

用一句更直白的话：

> 软约束管得住日常，硬约束管得住意外。

## 五、务实建议：该怎么选

不需要纠结"哪个更好"。更有用的问题是：**你现在的真实瓶颈在哪？**

### 如果你是日常写代码、改代码

用 CodeBuddy 就够了。通过 Skill 约束工作流，实现"写完 → 验证 → 失败修复 → 再验证"的基本闭环。

这已经能消灭大部分"宣称完成但没验证"的问题。

### 如果你想搭一个全自动流水线

比如输入一个需求，自动拆任务 → 写代码 → 跑测试 → 审查 → 修复 → 发布，中间不需要人工干预——那 DeerFlow 更合适，因为它能做到真正的多 Agent 编排和硬约束。

### 最务实的路线

1. **先用 CodeBuddy 的 Skill 体系把验证闭环跑通**——成本最低，见效最快
2. **观察哪些环节 Agent 总是跳步或失效**——这就是软约束的边界
3. **如果发现"软约束不够、Agent 总是跳步"，再考虑用 DeerFlow 把关键环节硬化**

顺序别反。先把纪律练熟，再考虑上系统。

## 六、常见误区

| 误区 | 为什么不对 | 更准确的理解 |
|------|------------|--------------|
| "CodeBuddy 能跑终端命令，所以它的验证和 DeerFlow 一样强" | 能跑命令 ≠ 能强制跑命令。CodeBuddy 的验证靠 Skill 文字约束，Agent 可以跳过 | CodeBuddy 是软约束验证，DeerFlow 是硬约束验证 |
| "DeerFlow 更强，所以应该直接上 DeerFlow" | DeerFlow 需要自己写 Python 编排，门槛高，不适合日常交互式开发 | 大多数场景下，CodeBuddy 的软约束已经够用 |
| "自动化验证就是跑一下测试" | 跑测试只是验证的一个环节。完整的验证还包括失败反馈、重试控制、熔断和经验持久化 | 自动化验证是一个闭环，不是一个动作 |
| "Skill 写了规则，Agent 就一定会遵守" | Skill 是建议，不是强制。上下文越长、任务越复杂，Agent 跳步的概率越高 | 关键流程不能只靠 Skill，要有外部验证兜底 |

## 七、一句话收束

自动化验证是 Harness Engineering 里最容易上手、也最容易被低估的一层。

它不需要你一上来就搭复杂平台。一个 Skill + 几条终端命令，就能形成基本闭环。但如果你的任务复杂度在持续上升，迟早会碰到软约束的天花板——那时候再考虑硬约束，也不迟。

关键不是"用哪个工具"，而是：

> **你的验证层，到底是在靠 Agent 自觉，还是靠系统兜底？**
