# Agent Team 的 Agent 该怎么划分：按职责、按阶段还是按能力？

> 很多人搭 Agent Team 的第一步是拆角色。拆完发现：6 个 Agent 各管一摊，但跑起来乱成一团。问题往往不在 Agent 不行，而在一开始就拆错了。Agent Team 的划分方式至少有三种——按职责、按阶段、按能力。选哪种，取决于任务结构，不是直觉。

前两天有人跟我说，他按照"产品、架构、前端、后端、测试、部署"拆了 6 个 Agent，结果跑起来前端 Agent 等后端 Agent 的接口定义，后端 Agent 又在等架构 Agent 更新数据模型，架构 Agent 还在等产品 Agent 确认需求变更——6 个 Agent 排成一条长长的依赖链，每个都在等上游，谁都动不了。他说："我以为拆成团队就是 Agent Team 了，没想到只是拆成了 6 个互相等待的串行阻塞。"

这个问题的根源不是 Agent 太多或太少，而是**划分方式和协调机制没对上**。6 个角色各自有依赖，但没有人管依赖的解耦和推进顺序——角色拆了，依赖关系没拆开，流水线就卡住了。

---

## 先给判断

赶时间就带走这三句：

- **按职责划分**——每个 Agent 对应一个专业岗位，适合分工明确、角色间依赖强的工程任务。
- **按阶段划分**——每个 Agent 包一个完整流程段，适合串行推进、阶段边界清晰的任务。
- **按能力划分**——每个 Agent 专精一种工具或技能，适合需要灵活组合、能力复用的场景。

三种策略没有绝对优劣。实际项目里最常见的做法是混合使用，但混合之前得先搞清楚每种策略在什么场景下成立。

---

## 一、按职责划分：最直觉，也最常见

### 什么是按职责划分

每个 Agent 对应一个专业角色，就像真实公司里的岗位分工：产品经理管需求，架构师管设计，前端工程师管页面，后端工程师管接口，测试工程师管质量。

搬到 Agent Team 里，就变成了：

| Agent | 对应职责 | 主要产出 |
|-------|---------|---------|
| Product Agent | 需求分析 | 需求文档、验收标准 |
| Architect Agent | 架构设计 | 接口定义、数据模型、技术选型 |
| Frontend Agent | 前端开发 | 页面、组件、样式 |
| Backend Agent | 后端开发 | API、业务逻辑、数据库操作 |
| Test Agent | 测试验证 | 用例、自动化测试、Bug 报告 |

### 案例：一条研发流水线

假设用 Agent Team 做一个"待办事项"Web 应用。按职责划分后，流程大概是这样的：

```text
Product Agent（输出需求文档 + 验收标准）
    ↓
Architect Agent（输出接口定义 + 数据模型）
    ↓
Frontend Agent + Backend Agent（按接口契约并行开发）
    ↓
Test Agent（跑测试、报 Bug、回归验证）
```

每个 Agent 只管自己那一摊。Product Agent 不写代码，Frontend Agent 不做需求分析。职责一收窄，每个 Agent 的 prompt 就可以做得更专业，上下文更集中，输出也更稳定。

### 优势

- **职责边界清晰**：每个 Agent 只需要懂自己的领域，prompt 容易写、容易调。
- **产出质量更高**：专注单一领域的 Agent 比"什么都干一点"的 Agent 输出更稳定。
- **接近真实团队**：拆分逻辑和人类团队一致，便于理解和维护。

### 劣势

- **角色间依赖强**：前端和后端必须对齐接口，需求变了所有人都得跟着改。没有协调机制就会打架。
- **协调成本高**：需要一个 Manager Agent 或协调者做版本对齐和上下文传递。角色越多，协调越难。
- **不适合小任务**：写个脚本、做个 demo，拆 5 个角色就是杀鸡用牛刀。

### 适合什么场景

| 适合 | 不适合 |
|------|--------|
| 前后端分离的完整项目 | 单文件脚本、小工具 |
| 有明确接口契约的工程任务 | 探索性调研、头脑风暴 |
| 需要持续迭代的项目 | 一次性 demo |

## 二、按阶段划分：一个 Agent 包一段

### 什么是按阶段划分

不按岗位拆，而是按流程的时间顺序拆。每个 Agent 负责一个完整的阶段，这个阶段内的所有工作都归它管。

类比工厂的工序：第一站负责原料处理，第二站负责成型，第三站负责质检。每一站的工人不需要懂其他站在干什么，只要把自己这一段做好、交给下一站就行。

### 案例：内容生产流水线

假设用 Agent Team 生产一篇技术文章。按阶段划分后是这样的：

| Agent | 阶段 | 产出 |
|-------|------|------|
| Scout Agent | 选题阶段 | 选题方案、角度建议 |
| Outline Agent | 大纲阶段 | 文章结构、章节安排 |
| Draft Agent | 初稿阶段 | 完整的 Markdown 初稿 |
| Review Agent | 审稿阶段 | 审稿报告、修改建议 |
| Polish Agent | 润色阶段 | 终稿 |

注意和按职责划分的区别：这里每个 Agent 不是按"专业能力"来分的，而是按"什么时候干活"来分的。Draft Agent 在初稿阶段可能既要写正文、又要配代码示例、又要画表格——它的职责是"把初稿阶段的所有事情做完"，而不是"只负责写作这一种能力"。

再看一个更偏业务的场景：客服工单处理。接单 Agent → 诊断 Agent → 处理 Agent → 复核 Agent，每个 Agent 负责一个完整阶段。处理 Agent 在自己的阶段里既要查数据库、又要调用退款接口、又要写客户回复——它管的是"这个阶段的一切"，而不是"某一种操作"。

### 优势

- **流程简单**：严格串行，A 做完交给 B，B 做完交给 C。不需要复杂的并行调度。
- **上下文传递清晰**：每个 Agent 接收上一阶段的完整产出作为输入，不需要从多个来源拼凑上下文。
- **阶段边界天然是质检点**：每两个阶段之间就是一个天然的门禁——上一阶段产出不合格，可以直接打回。

### 劣势

- **阶段粒度不好定**：拆太粗，一个 Agent 要干太多事，上下文可能爆；拆太细，阶段之间的传递成本就上去了。
- **阶段内职责太杂**：一个"开发阶段"的 Agent 可能既要写前端又要写后端，和单 Agent 全包的问题类似。
- **不利于并行**：严格串行意味着总耗时是所有阶段的累加。如果某些工作其实可以并行（比如前端和后端），按阶段划分就浪费了。

### 适合什么场景

| 适合 | 不适合 |
|------|--------|
| 串行流程清晰的任务（内容生产、文档处理） | 需要大量并行的工程任务 |
| 每个阶段内部复杂度可控 | 阶段内部需要多个专业角色协作 |
| 阶段间交接物明确（文档、文件） | 阶段间需要频繁回溯和协商 |

## 三、按能力划分：一个 Agent 管一种能力

### 什么是按能力划分

不按岗位拆，也不按流程阶段拆，而是按"能干什么"拆。每个 Agent 专精一种工具或技能，谁需要这个能力就调用它。

类比工具箱：扳手归扳手、螺丝刀归螺丝刀、电钻归电钻。拧螺丝就拿螺丝刀，打孔就拿电钻。工具不关心当前在做什么项目，它只关心"我能干什么"。

### 案例：研究型 Agent Team

假设用 Agent Team 做一份行业调研报告。按能力划分后是这样的：

| Agent | 能力 | 做什么 |
|-------|------|--------|
| Search Agent | 信息搜索 | 在网上搜索资料、论文、新闻 |
| Analyze Agent | 数据分析 | 整理搜索结果、提取关键数据、做对比 |
| Write Agent | 文本生成 | 把分析结果写成报告正文 |
| Cite Agent | 引用校验 | 检查所有引用是否真实、链接是否有效 |

DeerFlow（一个跑在 LangGraph 上的 Super Agent）的架构就接近这种思路：它有专门做搜索的节点、专门做代码执行的节点、专门做写作的节点。每个节点不是一个"角色"，而是一种"能力"。给 DeerFlow 一个调研任务时，Coordinator 会先调用搜索节点拿资料，再调用代码执行节点跑数据分析，最后调用写作节点生成报告——能力的调用顺序不是写死的，而是 Coordinator 根据当前任务动态决定的。下次换个任务，同样的搜索节点可能和翻译节点组合，完全不碰代码执行节点。这就是按能力划分的核心优势：**能力是积木，怎么拼看任务**。

### 优势

- **复用性强**：Search Agent 既能给调研任务搜资料，也能给编码任务搜文档。能力和业务解耦，换个场景也能用。
- **能力可组合**：新增一种能力（比如加一个翻译 Agent），不需要重新设计整条流水线，直接插进去就行。
- **单个 Agent 更简单**：每个 Agent 只做一件事，prompt 非常短，调试也容易。

### 劣势

- **业务逻辑分散**：没有一个 Agent 对整体业务负责。"先搜什么、再分析什么、最后写什么"的决策必须由调度层来做。
- **调度层复杂**：需要一个足够智能的协调者来决定"接下来该调用谁、传什么上下文"。这个协调者本身就很难写好。
- **上下文碎片化**：每个能力 Agent 只看到自己那一小块信息，可能丢失全局视角。

### 适合什么场景

| 适合 | 不适合 |
|------|--------|
| 需要灵活组合能力的研究/分析任务 | 流程固定、角色明确的工程任务 |
| 同一种能力被多个场景复用 | 每个任务的能力需求完全不同 |
| 有成熟的调度层（如 LangGraph） | 没有调度基础设施，纯靠 prompt 编排 |

## 四、三种策略对比

一张表看清楚：

| 维度 | 按职责划分 | 按阶段划分 | 按能力划分 |
|------|-----------|-----------|-----------|
| **划分依据** | 专业角色 | 流程阶段 | 工具/技能 |
| **典型拓扑** | 星型或网状 | 严格串行 | 调度层 + 能力池 |
| **上下文管理** | 通过契约对齐 | 阶段间交接 | 调度层统一管理 |
| **并行能力** | 天然支持（如前后端并行） | 弱，默认串行 | 强，调度层可动态编排 |
| **复用性** | 低——角色通常和项目绑定 | 中——阶段可跨项目复用 | 高——能力可跨场景复用 |
| **协调复杂度** | 高——需要管角色间依赖 | 低——线性传递 | 高——需要智能调度层 |
| **适合的任务** | 工程交付（前后端、测试） | 内容生产、文档处理 | 研究分析、灵活组合 |
| **常见坑** | 角色打架、接口对不上 | 阶段粒度拿不准 | 调度层写不好就失控 |

## 五、实战中最常见的做法：混合划分

纯用一种策略经常不够。真实项目里，最常见的做法是混合。

打个生活中的比方：一家餐厅的后厨。从"接单 → 备菜 → 烹饪 → 出餐"的主流程看，是**按阶段**划分的；但烹饪环节里有专门做热菜的、做冷菜的、做面点的，这是**按职责**划分的；而切菜、调汁、摆盘这些能力可以被热菜和冷菜共用，这是**按能力**复用的。一个正常运转的后厨，三种划分策略一直混着用。

### 案例 1：研发流水线中的混合划分

一个典型的 AI 研发 Agent Team，看着像按职责划分（Product、Architect、Frontend、Backend、Test），但仔细看：

- **Product → Architect → 开发 → Test**，这条主线其实是**按阶段串行**的
- **Frontend 和 Backend 并行开发**，它们之间的关系是**按职责划分**的
- **Manager Agent 全程协调**，它的角色更像是一种**调度能力**

所以实际上是：**主线按阶段、开发环节按职责、协调层按能力**。三种策略混在一起用了。

### 案例 2：DeerFlow 的混合划分

DeerFlow 的架构更明显：

- **搜索、代码执行、写作**——这些节点是按能力划分的
- **调研 → 分析 → 报告**——整体流程是按阶段推进的
- **Coordinator** 负责编排——它既是流程控制（阶段），也是能力调度（能力）

### 混合时要注意什么

混合划分最怕的是"混而不知"——以为自己在按职责拆，其实有些 Agent 是按阶段在干活，另一些是按能力在复用。搞不清楚用了什么策略，出了问题就不知道从哪调。

三条建议：

1. **先明确主策略**：Agent Team 主要按什么划分？先定一个主轴，其他策略是补充。
2. **画一张 Agent 关系图**：标清楚每个 Agent 的划分依据是什么。如果一个 Agent 既是"前端角色"又是"开发阶段的执行者"，至少心里要清楚它的双重身份。
3. **边界冲突时以主策略为准**：比如"这个功能到底该前端 Agent 做还是后端 Agent 做"，如果主策略是按职责，就按职责的边界来定；如果主策略是按阶段，就看它属于哪个阶段。

## 六、怎么判断你的场景该用哪种拆法

不想分析太多的话，跟着这三步走：

**第一步：看任务有没有明确的专业分工。** 比如前端、后端、测试各管一摊，接口契约清楚。如果有 → 按职责划分。

**第二步：如果分工不明确，看流程是不是严格串行的。** A 做完才能做 B，B 做完才能做 C，阶段间交接物清晰。如果是 → 按阶段划分。

**第三步：如果流程也不严格串行，看是不是需要灵活组合不同能力。** 搜索、分析、生成、校验自由搭配，同一种能力可能被多个任务复用。如果是 → 按能力划分。

**如果三步都走完还拿不准**，说明任务很可能需要混合划分——先按最核心的维度定主策略，其他策略往上叠。或者更简单：先按职责拆，跑通再调整。

不过在动手拆之前，还有一个更前置的判断值得先做：**这个任务真的需要拆成 Agent Team 吗？** 如果单 Agent 就能搞定，多拆几个角色只会增加复杂度，不会增加质量。写个脚本、做个 demo、改几行配置——这些任务拆 5 个 Agent 纯属自找麻烦。只有当任务复杂到"一个 Agent 撑不住"的时候，拆分才有价值。

## 七、常见错误与避坑清单

### 错误 1：拆太细——Agent 比任务还多

有人做一个简单的 CRUD 应用，拆了 8 个 Agent：需求 Agent、UI 设计 Agent、前端 Agent、后端 Agent、数据库 Agent、测试 Agent、部署 Agent、文档 Agent。

结果：Agent 之间传递上下文的开销比干活本身还大。协调者忙到崩溃，每个 Agent 干的活都不多，但串起来耗时极长。

**原则：Agent 的数量应该和任务的复杂度匹配，而不是和能想到的角色数量匹配。**

### 错误 2：拆太粗——一个 Agent 什么都干

另一个极端：把整个开发阶段塞给一个 Dev Agent。它既要写前端、又要写后端、还要配置数据库。上下文被三个领域的信息挤满，输出越来越不稳定。

**原则：如果一个 Agent 的 prompt 超过了一屏都写不完它该干什么，说明它该拆了。**

### 错误 3：只按直觉拆，不看上下文边界

很多人拆 Agent 的依据是"这个听起来应该是一个角色"。但 LLM 不是人，它的性能瓶颈是上下文窗口。

正确的拆法应该反过来想：**这个 Agent 的上下文需要包含什么信息？如果两个 Agent 需要的上下文高度重叠，也许它们应该合并；如果一个 Agent 需要的上下文来自完全不同的领域，也许它应该拆开。**

一个简单的判断方法：列出每个 Agent 必须读取的文件、文档和数据源。如果两个 Agent 的必读列表重合超过一大半，它们大概率应该合并成一个；如果一个 Agent 的必读列表里既有前端组件文档又有数据库 Schema 又有部署配置，它大概率应该拆开。

上下文边界，比职责边界更能决定 Agent 到底该怎么拆。

### 错误 4：划分了但没配协调机制

5 个 Agent 拆好了，但没有 Manager Agent，也没有统一的接口契约，也没有版本对齐机制。每个 Agent 自己跑自己的，结果合在一起全是冲突。

**原则：Agent 的划分和协调机制必须同时设计。拆了不管，比不拆更糟。**

## 结论

Agent Team 的划分不是拍脑袋决定的。三种主流策略各有适用场景：

- **按职责划分**适合分工明确的工程任务——职责边界就是 Agent 边界。
- **按阶段划分**适合串行推进的流程任务——阶段边界就是 Agent 边界。
- **按能力划分**适合需要灵活组合的分析任务——能力边界就是 Agent 边界。

真实项目里最常见的是混合使用。但混合之前必须搞清楚主策略是什么，否则出了问题不知道从哪调。而在所有划分策略之前，最值得先想清楚的一个问题是：这个任务到底需不需要拆。
