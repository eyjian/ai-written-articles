# MCP 服务和 Skill 到底是什么关系？一个管连接，一个管做事

> 很多人第一次接触 AI 智能体时，会把 `MCP` 和 `Skill` 混成一类：都像是在给 AI 加能力。但这两个词说的不是一回事。`MCP` 更像连接外部世界的通道，`Skill` 更像决定任务怎么做的流程。一个管“能不能连上”，一个管“连上以后怎么把事做对”。

难点就在于：它们经常一起出现。

你看到一个智能体会查数据库、会调接口、会自动执行任务，很容易顺手把这些能力都算到同一个筐里。可从架构上看，`MCP` 和 `Skill` 解决的根本不是同一层问题。

如果这两层不分开，后面就很容易越改越乱：明明缺的是连接层，你却在疯狂补 prompt；明明缺的是流程约束，你却不停加新工具。最后工具越来越多，智能体还是不稳定。

## 先给判断

先把结论放前面：

- **`MCP` 是连接层 / 通道 / 接口**：负责让 AI 摸到外部工具、API、数据库和服务。
- **`Skill` 是任务层 / 业务流程**：负责告诉 AI 这件事该怎么做、按什么顺序做、做到什么程度算完成。
- **外部工具、API、数据库** 才是真正执行动作的一层。

压成一句话就是：**`MCP` 决定“能不能连上”，`Skill` 决定“会不会做事”。**

## 为什么很多人会把两者混在一起

因为在实际使用里，它们常常是绑着出现的。

一个智能体去查订单、更新表格、发消息、调数据库，表面上看像是“它会做很多事”。但把动作拆开看，至少有两层：

第一层，是它**能接到这些外部能力**；第二层，是它**知道该先做什么、后做什么、什么时候停下来检查**。

前一层更像 `MCP`，后一层更像 `Skill`。

这也是为什么同样接了很多工具的智能体，有的看起来很稳，有的却像在乱按按钮。差别往往不在工具数量，而在有没有一套清楚的任务流程把这些工具管起来。

## 用最通俗的类比：把 AI 智能体当成外卖骑手

如果你想秒懂，可以把 **AI 智能体** 想成一个**外卖骑手**。

### 1. `MCP` 服务像骑手的“通信 + 接单系统”

它负责三件事：

- 连接外卖平台
- 连接商家
- 连接用户

它做的是**收发消息、调用接口、数据互通**。

但它不关心外卖好不好吃，也不负责决定骑手先去取餐还是先联系用户。换句话说，`MCP` 解决的是**接得上、连得通、调得到**。

### 2. `Skill` 像骑手的“送餐技能包 / 标准流程”

它负责的是另一套东西：

- 先确认订单信息
- 再决定取餐顺序
- 路上怎么规划路线
- 到达后怎么交付
- 遇到异常时怎么处理

这里面包含的不是连接协议，而是**任务目标、执行步骤、规则和判断逻辑**。

所以你可以把 `Skill` 理解成“怎么把这件事办明白”的工作方法。

### 3. 两者为什么不能互相替代

只靠 `MCP`，骑手当然能收到订单、能联系商家、能联系用户，但他未必知道怎么高效送达，可能会乱接、乱跑、乱回消息。

只靠 `Skill`，骑手即使知道一套很漂亮的送餐流程，如果手机没网、平台没连上、商家系统也接不到，他还是接不了单，更送不了餐。

所以它们从来不是谁替代谁，而是各管一层。

## 放回系统架构里看，就更清楚了

你给的这张文字架构图，已经把关系说得很直白了：

```text
┌─────────────────────────────────────────┐
│                 用户                      │
└───────────────────┬─────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│               LLM / 智能体                │
└───┬───────────────────────┬─────────────┘
    │                       │
    ▼                       ▼
┌──────────────┐      ┌──────────────┐
│    Skill     │ ←──→ │   MCP 服务    │
│ （业务逻辑） │      │ （连接协议）  │
│ 做什么、怎么做│      │ 怎么连、怎么调│
└───────┬──────┘      └───────┬──────┘
        │                     │
        │                     ▼
        │           ┌──────────────────┐
        └──────────→│  外部工具/API/DB  │
                    │ （真正干活的服务） │
                    └──────────────────┘
```

这张图里最关键的，不是 `Skill` 和 `MCP` 挨在一起，而是**它们职责不同**：

- **`LLM / 智能体`**：负责理解目标、做推理、决定接下来该干什么。
- **`Skill`**：把“该干什么”组织成一套更稳定的任务流程。
- **`MCP` 服务**：把“要调用外部能力”这件事接到真实世界。
- **外部工具 / API / DB**：真正执行查询、写入、调用、返回结果。

你也可以把关系理解成这样：**`Skill` 更像导演，`MCP` 更像电话线或调度台，真正演戏的是外部系统。**

## 一个最容易说错的地方：Skill 不是外部能力本身

很多人会下意识把 `Skill` 说成“AI 的能力包”。这话不能算全错，但不够准。

更稳的说法是：**`Skill` 是能力的组织方式，不是外部能力本身。**

比如“查订单并判断是否该退款”这件事里：

- 查订单，可能要连数据库或订单 API
- 查退款规则，可能要连知识库或规则服务
- 给出结论，才是流程判断

前两步要靠连接层，后一部分更像 `Skill` 在发挥作用。

所以 `Skill` 的强项不是“提供接口”，而是**把多个接口和判断步骤串成一个稳定流程**。

## 只有 MCP，没有 Skill，会发生什么

这种情况很常见：工具接了一堆，智能体也确实能调用，但执行过程像在碰运气。

常见表现是：

- 会调工具，但顺序乱
- 能查数据，但不会交叉验证
- 能写入系统，但不知道先检查风险
- 动不动就多调几次接口，成本高、结果还不稳

原因不复杂：它有“手”，但没有足够清楚的“做事方法”。

## 只有 Skill，没有 MCP，又会发生什么

这时候智能体往往看起来“很懂流程”，但它碰不到外部世界。

它可以把步骤讲得头头是道，也能生成一套像样的任务计划；可一旦任务需要查数据库、调接口、读外部服务、操作系统，它就卡住了。

简单说，**它知道怎么做，但摸不到东西。**

## 更稳的做法，是把两层分开看

下面这张表最适合拿来判断你到底缺哪一层：

| 你遇到的问题 | 更可能该补什么 |
|------|----------------|
| 智能体根本接不到数据库、API、浏览器、第三方系统 | 先补 `MCP` |
| 智能体已经能调工具，但顺序混乱、步骤跳跃、经常漏校验 | 先补 `Skill` |
| 智能体既接不到外部世界，又没有稳定流程 | 两层都补 |
| 只是做纯文本分析、写作、总结，不需要外部调用 | 可以先不碰 `MCP`，先看 `Skill` 或提示设计 |

最后一行尤其值得强调。

因为 **`Skill` 并不一定每次都要依赖 `MCP` 才能存在**。如果任务只是纯文本处理、纯推理、纯写作，它完全可以先在模型上下文里工作。

但只要你想让它**调用外部能力**，比如查数据库、调 API、发消息、读文件、跑服务，那它就离不开 `MCP` 这种连接层，或者至少离不开同类的工具接入机制。

## 什么时候该先想 MCP，什么时候该先想 Skill

如果你正在搭智能体，判断顺序可以很简单。

先问自己两个问题：

### 1. 它是不是要碰外部世界？

如果答案是“要”，先看连接层。

比如：

- 要不要查数据库
- 要不要调第三方 API
- 要不要操作浏览器或本地工具
- 要不要和其他系统交换数据

这些问题，本质上都先落在 `MCP` 这一层。

### 2. 它是不是已经会连了，但还是老翻车？

如果答案是“是”，就别继续堆工具了，先补流程。

比如：

- 总是没确认条件就直接执行
- 会查数据，但不会比对来源
- 写完结果就交差，不做验证
- 遇到异常没有固定处理顺序

这时候你缺的通常不是更多接口，而是一套更明确的 `Skill`。

## 最后记一个极简口诀

如果你只想带走一句最短的话，就记这个：

- **`MCP` = 能连上**
- **`Skill` = 会做事**

再补完整一点：

- **没有 `MCP`，AI 摸不到外部世界**
- **没有 `Skill`，AI 容易乱调用工具**
- **`Skill` 不能替代 `MCP`；一旦要调外部能力，`Skill` 反而要靠 `MCP` 这类连接层**

## 结论

把话收回来，`MCP` 和 `Skill` 不是一回事，也不该拿来互相替代。

`MCP` 解决的是连接问题，`Skill` 解决的是流程问题。一个让智能体接得上世界，一个让智能体做事别太乱。更稳的智能体，通常不是只把其中一层做强，而是把两层分清，再各自补齐。
